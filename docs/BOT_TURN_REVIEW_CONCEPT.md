ф# Концепт: «Разбор хода ИИ» (Bot Turn Review)

> Статус: КОНЦЕПТ на утверждение. Реализация НЕ начата.
> Заменяет row-by-row Bot Turn Theater как экран «Осмотреть ход».
> Дата: 2026-07-08.

---

## 1. Название

- **Режим / экран:** «Разбор хода ИИ» (в UI — заголовок «РАЗБОР ХОДА», подзаголовок «ИИ · Поколение N»).
- **Компоненты:**
  - чистая модель: `src/client/components/marsbot/botTurnReviewModel.ts` (PURE builder `MarsBotTurn → BotTurnReview`, unit-тестируемый под server-runner);
  - desktop: `BotTurnReviewOverlay.vue` (App-level, тот же слот, где сейчас `MarsBotTheaterOverlay`);
  - console: `ConsoleBotTurnReview.vue` (fullscreen overlay, паттерн `ConsoleColonyInspect`);
  - состояние: `botTurnReviewState.ts` (module reactive, наследник `marsBotTheaterState`, но БЕЗ таймеров проигрывания).
- Слово «theater» уходит из кода и текстов: это больше не проигрыватель, а review.

## 2. Как меняется роль theater

| Было | Становится |
| --- | --- |
| Live-проигрыватель: шаги появляются по таймеру (durationMs, MAX_TURN_MS=14000, live-подсветка, Skip, progress «9/10») | Мгновенный structured summary уже завершённого хода. Открылся → весь разбор виден сразу, ничего не «играет» |
| Лента строк в хронологическом порядке | Причинные группы (cause → effect), хронологическая лента остаётся ТОЛЬКО в раскрываемых «Подробностях» |
| Ощущение «бот делает ход сейчас» | Явное «ход завершён, вот что произошло и почему» |
| Единственный подробный слой | Третий слой поверх notification (слой 1: compact card; слой 2: журнал; слой 3: разбор) |

Notification-first flow НЕ меняется: compact card + staged visual commits остаются live-слоем. Разбор открывается:
- X «Осмотреть» на bot-turn notification (`expand-theater` → `open-review`);
- «Осмотреть ход» в журнале (`openMarsBotReplayByCorrelation` → тот же разбор);
- (опция, presentation-mode) `tm_bot_presentation='theater'` авто-открывает разбор при доставке карточки — контракт очереди/gate сохраняется.

## 3. Принцип приоритета информации — три уровня

1. **Уровень A (виден мгновенно, без скролла):** что разыграно + что реально изменилось (поле / параметры / ресурсы / жертвы) + итог.
2. **Уровень B (виден на экране, но визуально второй):** ПОЧЕМУ — причинные цепочки «метка → трек → действие → следствие».
3. **Уровень C (скрыт за раскрытием):** сырая хронология, внутренние клетки треков, тай-брейки, служебные строки.

Треки ИИ не прячутся — они подаются как ОБЪЯСНЕНИЕ (уровень B), а не как главный лог.

## 4. Общая структура экрана

Шесть зон (сверху вниз):

```
[1] HERO          — кто/когда + вердикт одной строкой + чипы итога
[2] КАРТА         — что разыграно (project / bonus), с правильной терминологией
[3] ЦЕПОЧКИ       — cause → effect map (метки → треки → действия → следствия)
[4] ПОЛЕ И МИР    — тайлы (показать на карте), глобальные параметры, достижения/награды
[5] ИГРОКИ И ИТОГ — атаки/задетые игроки + итог ИИ (before→after)
[6] ПОДРОБНОСТИ   — раскрываемая полная хронология (старые step-строки)
```

Зоны [4]/[5] рендерятся только если непустые. Зона [3] есть всегда (даже «метки без последствий» — это честная цепочка). Ход без карты (pass / failed без reveal) схлопывает [2] в строку внутри HERO.

### 4.1 HERO (вердикт)

- Строка идентичности: глиф бота + «ИИ» + сложность + «Поколение N».
- **Вердикт** — одно предложение, синтезированное моделью по приоритету:
  тайл > глобальный параметр > milestone/award > атака > track action > «продвинул внутренние треки».
  Примеры: «Разместил океан и получил 4 M€», «Торговал с Энцеладом через Судоходные линии», «Атаковал gg: −4 растения», «Продвинул треки — без изменений на поле».
- **Чипы итога** (макс ~5, приоритетные): глобальные параметры before→after (нейтральные, как `globalParamChips`), тайл-чип, TR-чип бота, самая крупная потеря жертвы, M€ бота. Это тот же визуальный словарь, что pills на notification — узнаваемость.

### 4.2 КАРТА (зона 2)

Два явных режима, различимых с первого взгляда:

**Project card** («РАЗЫГРАНА КАРТА ПРОЕКТА»):
- слева мини-карта (`JournalCardChip`-превью; клик/X → fullscreen через `CardZoomModal`/`consoleCardZoom`);
- справа плашка-объяснение правила ИИ (постоянная, короткая):
  «ИИ разыгрывает только метки карты — текст, стоимость и требования игнорируются»;
- строка меток: иконки top-right tags СЛЕВА НАПРАВО, каждая с мини-стрелкой к своему треку
  (`[🚀]→Космос  [🔬]→Наука  [⊘ Венера — дополнение не в игре]`);
- игнорированная метка — приглушённый чип `⊘` с причиной, не скрыта.

**Bonus card** («РАЗЫГРАНА БОНУСНАЯ КАРТА»):
- `BonusCardFace` (уже есть) с fate-бейджем (сбрасывается / уничтожается / каждое поколение);
- если у карты варианты a/b/c/d — подсветка ПРИМЕНЁННОГО варианта прямо на строках карты
  (применённая строка яркая + бейдж «применено», отклонённые — приглушены; причина отклонения показывается только если данные есть, см. §11);
- строка «Судьба карты»: «возвращается в колоду каждое поколение» / «уничтожена» / «сброшена».

### 4.3 ЦЕПОЧКИ (зона 3 — сердце экрана)

Не лента, а **колонка причинных карточек-цепочек**. Одна цепочка = одна причина + её следствия с отступом-коннектором:

```
┌ Метка [🚀 Космос] ────────────────────────────┐
│ Трек Космоса 4 → 5   ● достигнуто действие    │
│ └→ Разместить океан                            │
│    └→ океан на [ПОКАЗАТЬ]  · бонус клетки +2 M€│
│    └→ океаны 3 → 4 (глобальный параметр)       │
└────────────────────────────────────────────────┘
┌ Метка [🔬 Наука] ─────────────────────────────┐
│ Трек Науки 2 → 3   ● «Получить 3 РТ»          │
│ └→ ИИ получил 3 РТ                             │
└────────────────────────────────────────────────┘
```

Правила подачи:
- каскад «действие трека двигает другой трек, тот тоже срабатывает» — ВЛОЖЕННОСТЬ (глубина коннектора), а не соседние строки;
- движение трека БЕЗ действия — компактная строка «Трек X: 4 → 5» без раскраски-события;
- достижение действия — акцентный маркер ● + название действия (существующий `trackActionLabel`);
- skipped/ignored — приглушённая строка `⊘` внутри своей цепочки («не получает награду Гидросети — только финальные ПО», «метка дополнения — игнорируется»);
- потери/расходы ИИ показываются В цепочке-причине с подписью («стоимость торговли −1 M€», «тратит 1 деление трека Энергии»), никогда голой строкой «потерял 1 M€»;
- цепочки специальных причин: «Бонусная карта» (эффект bonus card), «Торговля с колонией», «Неудачное действие», «Гидросеть».

### 4.4 ПОЛЕ И МИР (зона 4)

- Ряд тайл-чипов: тип тайла + `[ПОКАЗАТЬ]` (journal-паттерн `boardCellHighlight`, пульс клетки);
- глобальные параметры before→after в стиле существующих нейтральных чипов; бонусные пороги шкалы (напр. «+2 M€ за океаны») — строка-следствие ПОД параметром, со ссылкой-причиной;
- milestone/award: престижная строка (золото/медаль, как в notification-вариантах) «ИИ заявил достижение X» + краткое «почему смог» (критерий выполнен — если строка лога есть).

### 4.5 ИГРОКИ И ИТОГ (зона 5)

Двухколоночный низ:
- **«Задеты игроки»** — по одной строке на жертву: цветной чип игрока → ресурс → `before → after` (красный loss-стиль `mb-imp__chip--loss`); исходы `MarsBotAttack.outcome` подаются явно: «растения защищены — карта уничтожена», «терять нечего», «цель выбирает, что потерять»;
- **«Итог ИИ»** — before→after чипы бота (M€ / TR / ресурсы / production) из `impact`-шага; компенсация failed action — с подписью причины.

### 4.6 ПОДРОБНОСТИ (зона 6, свёрнута по умолчанию)

- Кнопка «Полная хронология (N шагов)» → раскрывает СТАРЫЙ рендер `MarsBotTheaterStep` (строки в порядке событий) — компонент переиспользуется как есть;
- туда же: состояние всех треков ИИ (`MarsBotTracks` — уже готов), тай-брейк флипы («вскрыл карту X (стоимость 17), чтобы выбрать тайл колонии» — здесь слово «вскрыл» КОРРЕКТНО, это именно reveal для рандома), внутренняя бухгалтерия автомы.

## 6. Обязательные / опциональные блоки

**Обязательные (всегда):** HERO с вердиктом и чипами; зона цепочек (минимум одна); итог ИИ.
**Условные (если данные есть):** карта (нет при pass/чистом failed); поле и мир; задеты игроки; milestone/award.
**Раскрываемые (уровень C):** полная хронология; треки ИИ; тай-брейки; служебное.
**Никогда на главном слое:** внутренние клетки треков, причины детерминированных выборов без данных, debug.

## 7–13. Обязательные кейсы

1. **Project card, только метки** — карта + цепочка на метку: «Трек X: 4 → 5» без событий; HERO-вердикт «Продвинул треки — без изменений на поле»; зоны 4/5 (кроме итога ИИ) отсутствуют. Экран остаётся честным и коротким.
2. **Project card → track action** — метка-цепочка получает ● + вложенное следствие (тайл / параметр / РТ / floaters); board-следствия дублируются чипами в HERO и зоне 4.
3. **Каскад треков** — «● Продвинуть другой трек» → вложенная под-цепочка второго трека; если и он достиг действия — глубина 3. Коннекторы-отступы, а не плоские строки. Глубина визуально ограничена 3 уровнями (дальше — «→ см. хронологию», практический максимум автомы).
4. **Failed Action** — HERO-вердикт «Действие не удалось: …» (короткая причина из 10 enum-reason); одна цепочка «Неудачное действие»: причина полным текстом → «компенсация +5 M€ (3 на лёгком)»; итог ИИ показывает M€ before→after. Детали правила — в хронологии.
5. **Bonus card attack (Meteor Shower)** — карта с fate «уничтожается»; цепочка «Бонусная карта»: «атакует растения всех игроков»; зона 5 — построчно каждый игрок с outcome: hit (before→after красным), protected («растения защищены — карта Метеоритный дождь уничтожена» — престижно-заметно, это выигрыш игрока), nothing-to-lose, target-chooses.
6. **Bonus card с вариантами a/b/c/d** — на `BonusCardFace` применённая строка подсвечена + бейдж «применено»; отклонённые приглушены. Причина отклонения — ТОЛЬКО если сервер её даст (§11, Phase B); без данных — просто приглушение, без выдуманных объяснений.
7. **Tile placement** — зона 4: чип тайла + [ПОКАЗАТЬ] (пульс клетки `boardCellHighlight`); в цепочке-причине: бонус клетки (структурировано, а не англ. строка 'covered bonus icon(s)'), ocean adjacency «+2 M€ за соседний океан», параметр before→after. На консоли L3 = «показать на карте».
8. **Colonies / Shipping Lines** — цепочка «Бонусная карта · Судоходные линии»: правило карты («торгует с колонией, чей трекер продвинут дальше всех») как подпись-причина выбора; выбранная колония чипом (hover → `ColonyPreviewPopover`); «стоимость торговли −1 M€» с подписью; «хранилище Энцелада +2» с иконкой ресурса. Тай-брейк-флип («вскрыл карту … чтобы выбрать тайл») — в хронологию.
9. **Global parameter bonus chain** — параметр before→after + вложенное следствие порога («океаны достигли 3 → +2 M€», «бонусный шаг: дополнительное действие» → вложенная под-цепочка). Тот же коннектор-паттерн, что каскад треков.
10. **Milestone/Award** — престижная строка в зоне 4 (золотой/медальный акцент, `MaPreviewPopover` на hover); «почему» = критерий достижения (есть в статике достижений) + позиция ИИ, если строка лога это фиксирует.
11. **Delta Project / Hydronetwork** — данные УЖЕ есть (лог-строки AutomaDeltaProject): цепочка «Гидросеть»: «тратит 1 деление трека Энергии» → «продвигается на 1 ряд, достигает „Стандарты гидростроя“» → приглушённое ⊘ «награда этапа не выдаётся — ИИ получает только финальные ПО». Structured-поля не требуются для Phase A (лог-строки распознаются билдером по message-шаблону; Phase B может добавить `{kind:'delta'}` шаг для надёжности).

## 14. Console-native

- **Форма:** fullscreen overlay (не band у нижнего края — band был уместен для live-проигрывания, review хочет весь экран). Паттерн — `ConsoleColonyInspect`: тёмное стекло, крупные блоки, без мелкого лога.
- **Регистрация в leak-detector:** корневой класс (`.con-bot-review`) добавить в `SERVING_SURFACES` (урок `ConsoleGovernmentSupport` — иначе stranded-заглушка замаскирует экран).
- **Управление (хинты ТОЛЬКО в нижнем command bar):**
  - d-pad ↑/↓ — фокус по карточкам-цепочкам и секциям;
  - **A** — раскрыть/свернуть сфокусированную секцию (хронология, треки);
  - **X** — «Осмотреть карту»: project card → fullscreen `consoleCardZoom`; bonus card → крупный `BonusCardFace --large`;
  - **L3** — «Показать на карте»: overlay уходит в peek (полупрозрачный/скрыт), клетка пульсирует; любая кнопка возвращает разбор. Есть только при наличии тайла;
  - **B** — закрыть. Возврат туда, откуда открыт (нотификация уже погашена → игра; журнал → журнал).
- Открытие: X на bot-turn notification / X на журнальной записи automa-turn (существующие пути `openMarsBotReplay*` перенацеливаются).
- Пока разбор открыт — он holding-surface presentation flow (как театр сейчас): блокирует доставку нотификаций и mandatory-поверхности; закрытие дренирует очередь. Контракт `presentationFlow` не меняется, меняется только компонент.
- Типографика: заголовки Orbitron ≥15px, тело ≥17px (текущие console-нормы театра), фокус-стейт — яркая mint-рамка.

## 15. Desktop

## 17. Каких данных не хватает (и что добавить)

Реализацию предлагаю фазировать: **Phase A — клиент-only** (билдер группирует по ПОРЯДКУ шагов: tag-шаг «владеет» последующими advance до следующего tag; advance с action владеет последующими log/advance до следующего верхнеуровневого события; это уже даёт корректные цепочки для автомы, т.к. сервер пишет шаги строго в порядке разрешения). **Phase B — структурные поля** (убирают эвристику и открывают недостающие кейсы):

| Дыра (проверено — данных нет) | Предложение (Phase B, все поля optional/back-compat) |
| --- | --- |
| Явная связь «тег → эти advance» и «track action → каскад» | `MarsBotTurnStep.cause?: {kind:'tag', index} \| {kind:'track-action', trackIndex} \| {kind:'bonus-card'} \| {kind:'colony'} \| {kind:'failed'}` — штампуется в AutomaResolver/AutomaBonusCards, где причина известна |
| Выбранный вариант a/b/c/d бонусной карты + причины отказа | расширить reveal-шаг: `resolution?: {option: number, skipped?: Array<{option, reason: 'venus-maxed' \| 'no-colonies' \| …}>}` |
| Судьба бонусной карты В ЭТОМ ходу (сброшена/уничтожена/в колоду) | reveal-шаг: `fate?: 'discard' \| 'destroyed' \| 'recurring'` (сейчас только в AutomaState) |
| Структурный placement-бонус (бонус клетки / ocean adjacency / covered icons) | новый шаг `{kind:'placement', spaceId, tileType, bonuses: Array<{source:'cell-bonus'\|'ocean-adjacency'\|'covered-icons'\|'param-threshold', changes}>}` — сейчас это только англ. лог-строки |
| Правило выбора колонии (наиболее продвинутый трек / своя колония / флип) | лог-шаг о трейде дополнить `pickRule?: 'most-advanced-track' \| 'own-colony' \| 'flip'` — либо Phase A показывает ПРАВИЛО с bonus card (статично, детерминировано) |
| Сложность в самом ходе | `MarsBotTurn.difficulty` (для HERO; сейчас берётся из AutomaState — приемлемо и в Phase A) |
| Непереведённые строки | `'gained N M€ for N covered bonus icon(s)'`, `'Resources are protected'` → ru-ключи (Phase A, обязательный фикс) |

Phase A самодостаточна для утверждаемого UI; Phase B делает цепочки данными, а не эвристикой, и добавляет «применённый вариант» + структурные бонусы размещения. Порядок шагов сервера — контракт, который надо закрепить тестом (см. §20).

| «показал [карту]» (шапка notification) | «разыграл [карту]» | notification headline |
| «подробности хода» (turn details) | «Разбор хода» | header desktop+console |
| «ход завершён» (lingering) | остаётся ТОЛЬКО как статус в HERO, не как режим | — |
| «Бот думает» | уже удалён из модели — проверить греп-остатки в текстах/aria и добить | — |
| «Watch turn» / «Осмотреть ход» CTA | «Разбор хода» (или оставить «Осмотреть» — вопрос §21) | notification CTA |

| 'Resources are protected' | ru «Ресурсы защищены» | attack outcome |

Правило терминологии закрепляется: **«разыграл» = карта исполнена по правилам ИИ; «вскрыл» = только флип ради числа/тай-брейка.** Новые ключи — в `ru/automa.json`, с грепом на дубли (`make:json` падает на повторе).

## 20. Тесты и ручные проверки после реализации
