## Modern modal inputs (card-play sub-prompts)

When a played card (or any forced event) needs an answer, the server returns a `PlayerInput` as the TOP-LEVEL `waitingFor`. Those sub-prompts used to fall through to the legacy radio stack inside the hidden `.legacy-ui-overlay` (invisible unless the player manually opened Legacy UI). They now route to premium surfaces. `WaitingFor.shouldRouteToModal()` decides:

- The per-turn **action menu** (`or` titled `'Take your first action'` / `'Take your next action'`) is the ONE top-level `or` that stays inline (driven by the fork's dedicated action buttons). Every OTHER top-level `or` (card-driven OrOptions, WGT) routes to the modal.
- `MODAL_INPUT_TYPES` = `payment, option, player, amount, resource, resources, productionToLose, and, projectCard` — these can only ever be a mandatory sub-decision, so every instance routes to `MandatoryInputModal`. EXCEPTION for `projectCard`: `WaitingFor.useModalForCurrentInput` suppresses the modal when the top-level `projectCard` is routed to a dedicated overlay — `handPlayPrompt(view)` ("play a card FROM HAND" — EccentricSponsor / EcologyExperts via `PlayProjectCard`, candidates ⊆ hand) → КАРТЫ В РУКЕ overlay's PLAY mode; `standardProjectPlayPrompt(view)` ("play a standard project" — EstablishedMethods via `SelectStandardProjectToPlay`, candidates are standard projects NOT in hand) → Standard Projects overlay. BOTH reuse their overlay's exact existing pick → pay → submit flow; the only difference is the EMPTY response path (`playProjectCardAction` / `standardProjectsAction` return `{path: [], input}` for the top-level prompt, and `submitPlayCard` / `submitStandardProjectPayment` submit the BARE `{type:'projectCard', card, payment}`). The modal stays as a fallback only for a degenerate `projectCard` matching neither. Both overlays auto-open + minimize to the SHARED mandatory pill (`handPillVisible`/`handPillLabel`/`handPillTitle`, restore re-opens the owning overlay). The action menu's own `projectCard` is a NESTED option inside the top-level `or`, never top-level, so it's unaffected.
- Handled by dedicated surfaces OUTSIDE the modal (NOT routed here): `card` → `DraftFlowOverlay` / hand-select overlay; `space` → `PlacementBanner`; `colony` → `ColoniesOverlay`; `initialCards` → `InitialDraftFlowOverlay`; a FREE **award-funding** OrOptions → the modern **AwardsOverlay** in free-sponsorship mode (see below).

### Contextual choice modal (triggered-effect / on-play / deferred `OrOptions`)

A top-level `OrOptions` from a TRIGGERED effect (Pharmacy Union's science-tag "turn face down / do nothing", Olympus, Splice, Recyclon, Viral Enhancers, Mars University, Neptunian Power Consultants), an on-play decision, or a deferred action used to render via the bare `ModernOptionPicker` — a context-less "ВЫБЕРИТЕ ВАРИАНТ" list: no source card, no "why did this appear", weak result preview, no downside. The fix is the premium **`ContextualChoiceContent.vue`** (`src/client/components/modalInputs/`): a 2-column composition mirroring `CardActionConfirmContent` — the **source as a premium framed ARTIFACT on the LEFT** (`.contextual-choice__src-frame` dark-glass container + type-tinted glow rim + corner tick + hover zoom-hint; the card `:key`-ed, `autoTall`, click → fullscreen), a **kind chip + source name + framed "event summary" TRIGGER block** on the RIGHT, then the options below. The whole decision (source + summary + options) lives in ONE `.contextual-choice__scroll` area that scrolls AS A UNIT when it overflows — so a TALL card (corporations) is NEVER clipped inside a small fixed box (the earlier defect). It does NOT re-implement option handling: it WRAPS the (header-suppressed) shared `ModernOptionPicker` (`hideHeader` prop), so every interaction (nested inputs, board picker, hand pick, skip-split, disabled targets) is reused.

**Interaction model = ONE-CLICK, with inline-confirm only for RISKY options.** `ContextualChoiceContent` passes `:controlled="true"` (a SAFE leaf option commits on the FIRST click — no redundant footer confirm button; "Ничего не делать" / a plain gain are one-click) **plus** `:confirmRisky="true"`. In `confirmRisky` mode an option the server flagged RISKY — it carries a non-numeric `metadata.tradeoff` (irreversible side-effect, e.g. Pharmacy Union's "card turned face down") OR `warnings` — does NOT commit on the first click: it ARMS an inline **confirm DRAWER** attached under the SAME card (`.modal-input__option-card--confirming` squares the bottom + an amber `.modal-input__option-confirm` drawer with real Cancel/Confirm buttons — a SIBLING of the card, never nested, so it's valid HTML + keyboard-accessible; Esc disarms). A second click on the armed card also confirms. Both `controlled` + `confirmRisky` are OPT-IN props (default false) so every OTHER `ModernOptionPicker` consumer (WGT top-level select→confirm, the action/play-modal hosted steps) is byte-for-byte unchanged. The primary options are wrapped in a transparent `.modal-input__option-slot` flex item (so the drawer has a home); skip options stay in their own neutral block.

**Server marker (the only server change is additive + backward-compatible).** `BaseInputModel.choiceContext?: ChoiceContext` (`{source: {kind: 'card'|'corporation'|'standardProject'|'colony'|'system', card?}, trigger?, mode?: 'optional-effect'|'effect-choice'|'attack'|'reward'}`) — `src/common/models/PlayerInputModel.ts`. Set CO-LOCATED in the card via `OrOptions(...).markChoiceContext(...)` (`BasePlayerInput.markChoiceContext`), serialized CENTRALLY in `Server.getWaitingFor` (next to `startGamePrompt`/`awardFundingPrompt`). Builders are co-located-friendly: `src/server/inputs/choiceContext.ts` (`cardEffect(card, trigger, mode)` derives corp-vs-card from `card.type`; `attackEffect`/`effectChoice`/`systemChoice`/`namedCardEffect`). Per-option RESULT chips reuse `ActionEffect`: `OptionMetadata` gained `effects?` (rendered by the shared `ActionEffectChip`), `tradeoff?` (a NON-numeric downside shown as an amber warning — Pharmacy Union's "card turned face down"), and `description?`; chip builders in `src/server/inputs/optionMetadata.ts` (`chip`/`trChip`/`optionResult`). `ModernOptionPicker` renders all three on every option card (contextual or not).

**Routing:** `ModalInputHost.premiumComponent` returns `ContextualChoiceContent` for `type==='or' && choiceContext !== undefined`, else `ModernOptionPicker` (unchanged). A prompt WITHOUT the marker is byte-for-byte unaffected. Styles: `src/styles/contextual_choice.less` (+ `.modal-input__option-effects/-tradeoff/-desc` in `modal_inputs.less`). i18n: `ru/ui.json` (kicker/kind labels + trigger sentences). Playground: `?modalPlayground` has two `contextual`-tagged scenarios. Tests: `tests/models/choiceContext.spec.ts`, `tests/client/components/modalInputs/ContextualChoiceContent.spec.ts` + `ModalInputHost.spec.ts`.

**NOT a gap (already premium elsewhere):** blue-card `action()` branch picks render inside `CardActionConfirmContent` (the ДЕЙСТВИЯ overlay); on-play bespoke choices with a `cardPlayPreview` hook are pre-collected in the play modal. **Documented phase-2:** the shared deferred-action attack prompts (`StealResources`/`RemoveAnyPlants`/…) already have rich per-target metadata + disabled reasons; they only lack the source-card preview (needs threading the source `ICard` through the Executor). Full status: `docs/CHOICE_CONTEXT_AUDIT.md`. **To add a new contextual choice:** `.markChoiceContext(cardEffect(this, '<trigger i18n key>', '<mode>'))` on the `OrOptions` + per-option `.withMetadata(optionResult({effects, tradeoff}))`; add the trigger key to `ru/ui.json`.

### Free award funding → AwardsOverlay (not the generic option modal)

"Fund an award for free" (Vitor's start-of-game action) must NOT render as a generic `OrOptions` "pick an award" modal — it has a full premium AwardsOverlay already. The routing mirrors the hand-select / standard-project pattern:

- **Server marker, not title.** `BaseInputModel.awardFundingPrompt?: {free: boolean}` (server: `BasePlayerInput.markAwardFundingPrompt(meta)`, serialized centrally in `ServerModel.getWaitingFor` next to `startGamePrompt`). `Vitor.initialAction` builds its `OrOptions` with `.markAwardFundingPrompt({free: true})` AND each `SelectOption` titled with the **bare `award.name`** (was `'Fund X award'`) so the EXISTING `PlayerHome.findAwardOptionPath` (structure detection: all options' titles ∈ award names) + `submitInnerActionResponse(found, name)` (match option by title === name) machinery — already used by the PAID fund action nested in the action menu — works unchanged for the free top-level prompt.
- **Client state** lives in `src/client/components/awards/awardFundingState.ts` (module-level `reactive({active, minimized, signature})`, survives the playerkey remount). `freeAwardFundingPrompt(view)` = a top-level `or` with `awardFundingPrompt.free === true` (the paid action, nested in the action menu, never carries the marker, so it's never matched). `PlayerHome.awardFundingInput` watcher (immediate) enters the mode + auto-opens `activeOverlay='awards'` unless minimized, and exits when the prompt resolves.
- **Suppress the generic modal:** `WaitingFor.useModalForCurrentInput` returns `false` when `freeAwardFundingPrompt` matches (mirror of the corp suppression).
- **AwardsOverlay free mode:** a `freeFunding` prop adds a mint "БЕСПЛАТНОЕ СПОНСИРОВАНИЕ" banner + hint, shows the cost as `Free`/`0` (`effectiveCost` computed), and drops the M€ affordability check in `fundBlockerReason`. The fund button already reads "Спонсировать" (the `Fund` key — the overlay was ALREADY consistent; the generic modal's `play_prompts.json` "Профинансируйте…" was the inconsistency, now bypassed). `fundAward(name)` rides the existing path and `exitAwardFunding()`s on submit (no pill flash).
- **Mandatory ⇒ minimize, never dismiss.** `minimizeMandatoryHandPrompts` / `toggleOverlay` (now includes `'awards'`) / `onAwardsOverlayClose` minimize to the SHARED mandatory pill (`handPillVisible`/`handPillTitle` include award-funding; `restoreHandPill` re-opens awards). Title in the pill: "Free sponsorship". New ru keys (`ui.json`): `Free`, `Free sponsorship`, `Choose an award to sponsor for free.`. (Asimov CEO's free funding is OUT of scope — its awards aren't in `game.awards` yet, so the overlay can't render them; it stays on the generic modal.)

Inside the modal, `src/client/components/modalInputs/ModalInputHost.vue` is a premium-first router: it renders a modern component where one exists and FALLS BACK to the legacy `PlayerInputFactory` otherwise, so a not-yet-migrated type is still visible (inside the modal) rather than buried. The premium components (all submit byte-identical payloads to their legacy `Select*.vue` twin — the legacy components are kept for the inline path, per "don't refactor the radio stack"):

- `ModernOptionPicker` (`or`) — generalises `WorldGovernmentModalContent`: a leaf `option` commits instantly; a `space` option arms board picker-mode (modal fades, `PlacementBanner` shows); any other nested input expands into a wizard step hosted recursively via `<modal-input-host>` (registered GLOBALLY in `main.ts`, same trick as `<player-input-factory>`, to break the `ModalInputHost ↔ ModernOptionPicker` type cycle that collapses vue-tsc to `{}`). Preserves the learner-mode option filter + original-index mapping.
- `ModernConfirm` (`option`), `ModernPlayerPicker` (`player`), `ModernAmountSelector` (`amount`), `ModernResourcePicker` (`resource`), `ModernResourcesPicker` (`resources`), `ModernProductionToLose` (`productionToLose`).
- `CardSelectionContent` (`card`) — the SAME premium card grid `DraftFlowOverlay` uses for top-level SelectCard. It is reached here when a card option is **NESTED inside a modal-hosted `OrOptions`** and `ModernOptionPicker` expands it into a wizard step via `<modal-input-host>`: e.g. AstroDrill "add an asteroid to ANY card", ImportedHydrogen "add 3 microbes / 2 animals to a card", GrantResource/GrantVenusAltTrackBonus/GainAnyResourceButScience "add a resource to a card", RemoveResourcesFromCard, **Mars University "discard a card to draw"**, and the other ~17 nested SelectCards. Before this, those fell to the legacy `SelectCard.vue` checkbox/radio list with a purple `Добавить …`/`Сбросить` button (the legacy card-choice UI this fork is replacing) — `.legacy-ui-overlay` is `display:none`, so the ONLY way that legacy list ever became visible was the `ModalInputHost` factory fallback for `'card'`. `CardSelectionContent` submits the byte-identical `{type:'card',cards}`, which `ModernOptionPicker.nestedSave` wraps into the outer OR (`{type:'or',index,response:{type:'card',cards}}`) — no server change. `ModernOptionPicker` drops the redundant `.modal-input__nested-label` for a card step (the grid shows the SelectCard title itself) and adds `modal-input--wide-nested` so the host `.modal-input` sheds its 420–720px clamp; the modal CARD already widens via the existing `:has(.card-selection)` rule (matches the descendant grid at any depth), so the grid sizes to the viewport exactly as in `DraftFlowOverlay`. `CardSelectionContent.actionLabel` surfaces the server's verb buttonLabel on a forced single-select (`Добавить астероид` / `Сбросить` / `Удалить ресурс(ы)` / `Показать` — the same verb the legacy submit button showed) instead of a generic `ВЫБРАТЬ`, EXCEPT for draft picks (`isDraftPhase`, buttonLabel `Keep`) and the generic `Save`/`Select`/`Ok`/`''` labels, which stay `ВЫБРАТЬ` for consistency. **Routing rule (unified):** a TOP-LEVEL hand SelectCard → КАРТЫ В РУКЕ overlay; a TOP-LEVEL non-hand SelectCard → `DraftFlowOverlay` → `CardSelectionContent`; a NESTED SelectCard (inside a modal OrOptions) → `ModalInputHost` → `CardSelectionContent`. So a nested discard (Mars University) shows the premium modal grid rather than the hand overlay — re-routing a nested prompt to the hand overlay would mean threading the OR-wrapping through PlayerHome, which the "don't spread if/else" rule forbids; the premium grid is the consistent surface for nested card picks. **KNOWN narrow gap:** a SelectCard nested inside a top-level `AndOptions` (only `FocusedOrganization` [prelude2] + `AeronGenomics` [underworld] — "discard 1 card AND spend 1 resource" / "pick a card AND pick tokens") still renders via the legacy `AndOptions.vue` (`'and'` has no premium component). A premium `ModernAndOptions` was deliberately NOT built: routing `'and'` would also hit the bespoke colony-trade AndOptions (`ColonyTradePaymentModal`) and 5 other resource-combo AndOptions, and the premium child inputs are self-submitting (no "controlled" mode), so a faithful combined-submit replacement is a larger, riskier change than the 2 niche cards justify. Revisit as a focused task.

**Premium visual language (polish pass) — benchmark = the colony-trade modal.** The shared `.modal-input__*` styles were lifted to the `ColonyTradePaymentModal` level so every modern input reads as one design system, not a generic form. Touch `modal_inputs.less`, not per-component ad-hoc styles:
- **Primary CTA** (`.modal-input__primary-btn`) is now the single strongest control — a bright cyan plate + dark glyphs + glow (mirrors `.colony-trade-pay__confirm`), was a weaker solid-green outline. Disabled stays OPAQUE (flat dark plate + muted text, never see-through), per the project rule.
- **Secondary buttons** (`.modal-input__back-btn` / `__max-btn`) are calm dark-glass with a cyan rim + hover lift — quieter than the CTA but clearly from the same kit. The MAX button shows the value (`МАКС 5`) via `.modal-input__max-btn-value`.
- **Option / target / resource cards** share an 8px radius + hover-lift + cyan-aura selected state.
- **SelectResource grid** is a CENTRED `flex-wrap` of fixed-width (108px) reward tiles (was `repeat(auto-fit, minmax(96px,1fr))`, which stretched tiles and left a 5+1 orphan row) — 5 reads as a deliberate 3+2.
- **SelectResources / ProductionToLose** rows are restructured into a left IDENTITY area (`.modal-input__dist-id`: icon + name) + a grouped stepper CLUSTER (`.modal-input__dist-controls`), light up cyan when allocated (`--active`), and the running total is a premium centred chip (`.modal-input__dist-counter-text`, mint+glow when complete). ProductionToLose frames its icon in `.modal-input__prod-frame` (brown production tile).
- **SelectAmount** central value is a framed control-panel readout (`.modal-input__step-readout`, 40px value).
- **Skip / neutral options** ("Do not remove plants", "Do nothing" — `metadata.kind === 'skip'`) are PULLED OUT of the target list (`ModernOptionPicker.primaryEntries` vs `skipEntries`, preserving the displayed index that `pickOption`/`selectedIdx`/`data-test` key off) into a SEPARATE bottom block (`.modal-input__skip-options`, dashed accent-less calm card + `⊘` glyph) — a safe fallback never sits BETWEEN two player targets again. Detection is the server marker `kind`, never title text.

To add a premium input for a new type: build the component (mirror an existing one's submit shape), add it to `PREMIUM_COMPONENTS` in `ModalInputHost.vue`, and (if the type isn't already covered) add it to `MODAL_INPUT_TYPES` in `WaitingFor.vue`. Styles live in `src/styles/modal_inputs.less` (`.modal-input__*`, `@import` in `common.less`).

### Option UI metadata (rich choice cards)

A generic `OrOptions` only carries text labels, so the client can't render an icon / player chip / impact preview without semantic context. The OPTIONAL `OptionMetadata` (`src/common/models/PlayerInputModel.ts`, on `SelectOptionModel.metadata`) carries that context — `{kind, icon, amount, player:{color,current,resulting}, global:{current,resulting,unit}}` — **alongside** the text, never replacing it. Backward-compatible: an option with NO metadata renders via the text fallback (the previous behaviour). Server side, `SelectOption.withMetadata(...)` attaches it (serialized in `toModel()`), and `src/server/inputs/optionMetadata.ts` has factory helpers so cards declare intent without boilerplate — add a factory per recurring pattern rather than hand-building metadata at the call site. Current helpers:

- `removeResourceFromPlayer(target, resource, amount, current)` / `stealResourceFromPlayer(...)` — destructive player-target (kind `resourceRemoval` / `steal`), with a `current → resulting` preview on the target.
- `gainResource(resource, amount)` — standard-resource gain (kind `resourceGain`).
- `globalParameter(icon, steps, current, resulting, unit?)` — raise a global parameter (kind `globalParameter`); `icon` ∈ `temperature` / `venus` / `oxygen` / `ocean`, `unit` is the symbol (`°C` / `%`), preview reads `global.current → global.resulting`.
- `addResourceToCard(cardResource, amount?)` / `removeResourceFromCard(cardResource, amount?)` — card-resource add/remove (icon = the `card-resource-<key>` suffix, derived from the `CardResource` value by lowercasing + spaces→hyphens; correct for every in-scope resource, only the out-of-scope fan card `RESOURCE_CUBE` differs).
- `skip()` — a "do nothing / skip" option (kind `skip`).

**Icon families are resolved in one shared place: `src/client/components/modalInputs/optionIcons.ts` `iconClassFor(icon)`.** It maps an icon-key to one of three CSS class families — standard resources → `resource_icon resource_icon--<key>`, global parameters → `wgt-icon wgt-icon--<key>`, everything else → `card-resource card-resource-<key>`. `ModernOptionPicker` (lead icon + preview icon) and `ModernAmountSelector` both call it, and `modal_inputs.less` normalises the size of all three families via the shared `.modal-input__option-icon` class (the `wgt-icon` set is otherwise only sized inside a compound selector, so the standalone normalisation is required). When you introduce a new icon-key, put it in the right set in `optionIcons.ts` — don't hand-build the class string at the call site.

Client side, `ModernOptionPicker` reads `optionMeta(opt)` and renders the rich card (player colour dot + name, the resolved icon, `current → resulting` preview when either `player` or `global` carries both values); `optionColor` prefers `metadata.player.color` and falls back to a PLAYER token in the title Message; `optionActionText` uses the `buttonLabel` verb for player-target options (player + amount already live in the chip + preview) and the descriptive title otherwise.

**`SelectAmount` carries the same icon vocabulary.** `SelectAmountModel` has optional `icon?` / `unit?`; the server `SelectAmount` constructor takes an optional 6th `options?: {icon?, unit?}` (backward-compatible — omit for a bare stepper). `ModernAmountSelector` renders the icon + unit beside the value via the same `iconClassFor`.

**`SelectPlayer` is a PREMIUM TARGET PICKER, not a name list (`ModernPlayerPicker`).** Each candidate is a rich glass *target card*: colour accent bar + dot, name, corporation (looked up client-side via `getCard(card.name).type === CardType.CORPORATION` over the public `tableau`), and — when the server supplies `{icon, amount, scope}` — a **per-target impact mini-HUD** (`current → resulting`, with a "ПРОИЗВОДСТВО" label over the framed icon for production). The header is a short heading + lighter subtitle (the full server instruction) + a secondary HUD effect chip; the flow is **select → confirm** (click selects, a premium CTA commits) so a destructive pick is never a single mis-click. `SelectPlayerModel` carries optional `icon?` / `amount?` / `scope?: 'stock' | 'production'`; the server `SelectPlayer` constructor takes an optional 4th `options?: {icon?, amount?, scope?, disabled?}`. **`scope` is the production-vs-stock distinction** — `'production'` reads each player's `*Production` field and draws the icon inside a brown "production frame" (`.modal-input__prod-frame`), `'stock'` (default) reads the plain resource field. Per-target current/resulting are computed **client-side** from the public player models (every player's stock + production is public), so the server only sends the hint — no per-target payload.

**State semantics are kept visually distinct** (CLAUDE goal: never make a player guess): SELECTED = the one strong cyan accent; SELF = a *local* amber "Это вы" chip ONLY (never a whole-card highlight, so self ≠ selected); DISABLED = subdued + non-interactive but still shows its reason; muted = a `resulting === current` no-op (soft, still clickable).

**Unavailable targets are SHOWN, not hidden.** Instead of silently dropping a relevant-but-unpickable player, the server passes them via `SelectPlayer`'s `disabled: ReadonlyArray<{player, reason}>` → `SelectPlayerModel.disabledPlayers` → greyed, non-selectable cards with the reason. `DecreaseAnyProduction` surfaces opponents who can't be reduced (`Production already at minimum` / `Production is protected`, mirroring `Player.canHaveProductionReduced`); `CometForVenus` surfaces opponents with `No Venus tag`. The selectable `players` list (what the server validates against) is unchanged — disabled entries never submit. Reasons are English i18n keys translated client-side (`ru/ui.json`). Wired (impact): `DecreaseAnyProduction` (shared → Energy Tapping & every production-attack card, `scope: 'production'`), `Flooding` / `CometForVenus` / `LawSuit` (`scope: 'stock'`, M€). A bare `SelectPlayer` with no icon (e.g. Colony "discard a card") still renders rich name+corp cards.

**`OrOptions` carries the same disabled-target idea via `OrOptionsModel.disabledOptions` (`DisabledOptionModel[]`).** The server `OrOptions.setDisabledOptions(...)` attaches informational, non-selectable entries; `ModernOptionPicker` renders them under an "НЕДОСТУПНЫЕ ЦЕЛИ" heading as greyed twin cards (player chip + greyed icon + reason). Built via the `disabledPlayerTarget(target, icon, reason)` helper in `optionMetadata.ts`. Wired: `RemoveAnyPlants` (`No plants to remove` / `Plants are protected`), `StealResources` (`Nothing to steal` / `Resources are protected`), `Sabotage` + `HiredRaiders` (`No resources to remove` / `Resources are protected`). Never processed — `disabledOptions` is a separate field, so option indices (and `OrOptions.process`) are unchanged.

**Colony TRADE flow is a confirm-first premium modal (`ColonyTradePaymentModal`).** Pressing **ТОРГОВАТЬ** on a colony NEVER trades instantly — `PlayerHome.onColonySelected` (trade mode) ALWAYS opens `ColonyTradePaymentModal` (even with one pay path), and the server action fires only on the modal's **ПОДТВЕРДИТЬ ТОРГОВЛЮ** (select → confirm). The colony-tile action button is mode-aware: `ColonyTile`'s `mode` prop → `selectLabel` renders `Trade` in trade mode, `Select` for SelectColony. The modal shows a colony summary (planet + name + the reward at the current track position via `BenefitGlyph`/`tradeAtIndex`) and every payment path as a rich option card (resource icon + cost + `In stock: current → resulting`). UNAFFORDABLE standard resources are shown DISABLED with a reason instead of vanishing: server `Colonies.tradeWithColony` adds them to the inner `OrOptions.setDisabledOptions(...)` (via each `IColonyTrader`'s new optional `optionMetadata()` + `disabledReason()`), and the bespoke modal renders `payOr.disabledOptions` greyed (`data-hint` tooltip). The self-resource preview rides a NEW `OptionMetadata.resource: {current, resulting}` field. The trade-payment OrOptions is the `'Pay trade fee'` inner OrOptions of the `'Trade with a colony tile'` AndOptions — `PlayerHome.findTradeColonyContext` extracts both `paymentOptions` and `disabledPayments`. (Localization gap fixed: `'Pay ${0} energy'` had no ru key — the energy option rendered untranslated "PAY 3 ENERGY"; added to `ru/ui.json`.) The bespoke modal is in `?modalPlayground` (a dedicated section, since it isn't routed via `modal-input-host`) so colony modals stay in the audit gate.

**Trade-bonus transparency block.** When colonies are already built on the tile, trading hands every colony owner the FIXED `metadata.colony` bonus (a SEPARATE, track-independent reward — Luna 2 M€, Ceres 2 steel — NOT the trade reward). `ColonyTradePaymentModal` surfaces this as a SECONDARY block under the colony summary (`.colony-trade-pay__bonus`, quieter than the summary so it never steals focus from the payment decision): the bonus benefit once via `BenefitGlyph` (`colonyBonus` computed = `getColony(name).colony`, built like `tradeReward`) + a player chip per owner with an explicit `×N` when one player holds several colonies here. NO server change needed — the client already has `colony.colonies` (`Array<Color>`, duplicates = multiple colonies) and the full metadata via the client `getColony` manifest; the modal only gained a `players` prop (`playerView.players`) to name them. Shown ONLY when `beneficiaries.length > 0`; includes the trading player (they DO get the bonus if they have a colony here). ru key: `Trade bonus to colonies here`.

**Colony-trade PRE-SELECT (the shared preview layer — desktop modal + console composer).** The trade confirm follows the play-modal/blue-action philosophy: EVERY decision is made BEFORE one final submit. The shared source of truth is a bounded READ-ONLY endpoint **`GET /api/game/colony-trade-preview?id=<playerId>&colony=<name>`** (`ApiGameColonyTradePreview` → `src/server/colonies/colonyTradePreview.ts` `buildColonyTradePreview`) returning `ColonyTradePreviewModel` (`src/common/models/`): the track math mirrored from `Colony.trade` (`{current, effective, steps, willAsk}`), `rewardQuantity` at the effective position, `megacreditsPayment` (the REAL `SelectPaymentDeferred.previewPaymentModel()` — set ONLY when the M€ path would prompt; cost via the new `TradeWithMegacredits.cost` getter so Adhai/discount math never forks), and `followUps` — the trading player's OWN post-submit prompts **in live deferred-queue order**: `trackChoice` (an `ask` colony + offset) → own `colonyBonus` card-targets (GiveColonyBonus defers at DEFAULT — BEFORE the reward's AddResourcesToCard at GAIN_RESOURCE_OR_PRODUCTION) → the `tradeReward` target. A `cardTarget` follow-up comes from the REAL `AddResourcesToCard` (`getCards`/`previewSelectCard`): ≥2 candidates → `pick` (pre-collectable SelectCardModel), exactly 1 → explicit `auto` (the live path auto-applies — shown, never silent), 0 → `lost: true` (honest "resource is not added" warning). Non-collectable interactive rewards (steal / opponent-discard / draw-and-keep / copy-trade / ocean / delegates / WGT-ocean) surface as `note` follow-ups → "after confirming: …" lines. **`SelectColony` carries it via `SelectColonyModel.disabledColonies` + a `purpose` discriminator.** Two modes: `purpose: 'selectExistingColony'` (trade / build on an existing tile) shows **every in-play colony** (`game.colonies`) with the unpickable ones DISABLED + a reason; `purpose: 'addNewColonyToGame'` (Aridor — add a discarded tile) shows ONLY the offered (not-in-play) tiles, never the existing ones (criterion: don't list existing colonies for an add). The server `SelectColony` has public `purpose` + `disabledColonies: [{colony, reason}]` fields (set after construction, like the existing `showTileOnly`); `toModel` serialises disabled to `{name, reason}`. `BuildColony` computes the disabled in-play colonies (game colonies ∉ `getPlayableColonies`) and names each reason it can't derive client-side — `Colony is full` / `You already have a colony here` / `Cannot afford the TR increase to build here` (the Venus/Europa/Leavitt TR-affordability rule the client can't compute; the affected parameter is obvious from the tile, so one message covers all three). **`SelectCard` carries it via `SelectCardModel.disabledCards` (a SEPARATE list of `CardModel`s, NOT mixed into `cards`).** The server `SelectCard` constructor takes `disabled?: ReadonlyArray<{card, reason}>` in its config; `toModel` serialises them through `cardsToModel(…, {enabled: all-false, disabledReasons})` so each disabled `CardModel` gets `isDisabled = true` + `disabledReason` (new `CardModel` field). Keeping them OUT of `cards` is deliberate: `cards` stays the selectable set, so `SelectCard.process` (which validates membership in `this.cards`) rejects them for free, auto-select/count logic is unchanged, and existing tests that assert `cards.length` don't break. `CardSelectionContent.vue` merges the two for display behind an **All / Available / Unavailable filter** (default **Available**, only shown when `disabledCards` is non-empty); a disabled card renders greyed with a muted reason chip (`data-hint` premium tooltip for the full text) instead of an action button. **Scope is case-specific, not a count heuristic** — the deferred decides what's "relevant". `RemoveResourcesFromCard.getUnavailableTargetCards` enumerates the SAME resource-type cards (`getResourceCards(resourceType)`, regardless of count) that aren't selectable and tags each `No resources on this card` (count 0) or `Card resources are protected`. All such cards are PLAYED/tableau cards (public) — no privacy leak. Only wired for a specific `cardResource` (the "cards that can hold X" pool); an untyped removal stays as-is.
