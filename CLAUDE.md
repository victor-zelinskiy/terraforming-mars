## Project Goals (vize1215 fork)

This is `vize1215`'s personal fork — a private/self-hosted build of the open-source `terraforming-mars` project. The active UI work is driven by these goals; weigh decisions against them when proposing or making changes:

1. **Single-screen, no-scroll play.** The active game state (board, current player's resources/tags, hand, played-cards filter, etc.) should fit on screen at typical desktop resolutions. Long pages with vertical scroll are the thing we are getting rid of — content that doesn't fit becomes an overlay or a panel, not a scroll target. The top/bottom bar-button overlays (`bar-overlay--*`) are the canonical mechanism for "secondary" content (Log, Played cards, Milestones, Awards, Standard Projects, Colonies, Cards).
2. **Steam-version visual feel.** The reference for layout density and proportion is the Asmodee Digital Steam release of *Terraforming Mars*: large central board, compact peripheral chrome, board scaled up to dominate the screen as more vertical space is freed.
3. **Ark Nova (BoardGameArena) animation feel.** When choosing animations / transitions / hover effects, lean toward the smoothness of BGA's Ark Nova implementation — short easings, subtle scale/glow on interactive elements, no hard pop-ins. Don't introduce flashy motion just because something is being changed; default to calm.
4. **Active visual refresh.** This fork explicitly wants the game to *feel* more modern than upstream. When touching UI, take the visual freedom to refresh things: sci-fi typography for panels/labels (Prototype, Orbitron-style families that ship with the repo) where the upstream choice is generic; subtle gradient/clip-path borders on grouped blocks (resource clusters, tag clusters) to give a control-panel vibe; consistent dark glassmorphic backgrounds for floating panels. The user will push back if a specific change misses, but the default is "try to improve the look" rather than "match upstream pixel-for-pixel". When in doubt, prefer the most polished option that still respects goals 1–3.

When a change has trade-offs between these goals and any other consideration (closeness to upstream, code volume, edge cases), favor the goals above unless the user says otherwise.

## Action UI Rework (in progress)

This fork is migrating the per-action UI away from upstream's generic `wf-action` + `btn-submit` radio-button form and toward **dedicated styled buttons** on each game element (milestones first, awards / standard projects / convert-plants / colonies to follow). The radio UI still exists and works — once every action type is migrated it will be hidden, not removed.

When you add a new dedicated action button, follow this contract:

1. **Tie the button to the server's action-availability logic, not to the radio render.** Walk the `playerView.waitingFor` tree (recursively, since the prompt can be nested) to find the `OrOptions` whose `title` matches the action prompt (e.g. `'Claim a milestone'`, `'Convert 8 plants into greenery'`). The server has already filtered options by every rule (cost, prerequisites, phase, opponents-have-passed, etc.) — **the option's presence in that tree IS the source of truth for "available right now."** Do not re-derive availability from raw player state on the client.
2. **Submit through `WaitingFor.onsave()`** with the nested `OrOptionsResponse` payload (`{type:'or', index:I, response:{...}}` wrapping recursively, with a `{type:'option'}` innermost). Bypasses the radio UI but is byte-for-byte identical to what `OrOptions.vue` would have sent — no server changes needed. The reference implementation lives in `PlayerHome.vue` → `findMilestoneOptionPath` + `claimMilestone`.
3. **Show the button as disabled, not hidden, when the player meets the conceptual prerequisite (e.g. score threshold) but can't act right now.** Use a `:title` tooltip with the blocker reason translated via i18n. Reasons we distinguish for milestones: insufficient M€ (`Not enough M€`), and not-your-turn / mid-sub-action (`Not your turn to take any actions`). Hiding the button is worse UX — the user is left guessing why an action they're entitled to disappeared.
4. **Don't refactor the existing radio-button stack** (`WaitingFor.vue`, `OrOptions.vue`, `SelectOption.vue`, server prompt code). They have to keep working unchanged while migration is in progress, and the eventual hide is a stylesheet flip — not a deletion.

The Milestones overlay (`MilestonesOverlay.vue` + `PlayerHome.vue` claim handlers) is the canonical example; mirror its detection + submission pattern when you wire up new action buttons.

## UI Philosophy: dedicated buttons vs. mandatory-input modals

This fork is moving toward a layout where the **Actions** section in the player home no longer exists as the catch-all bin for everything the server is waiting on. The goal is single-screen no-scroll play (CLAUDE.md goal #1), and the legacy "scroll from board to actions, pick a radio, hit submit" loop is the single biggest blocker to that.

The replacement model has two distinct kinds of prompts and a different UI surface for each:

**A. Standard top-level actions** — things the player can voluntarily choose to do on their turn from the regular action menu (`Player.getActions()` in `Player.ts`). The full list:
- Play a project card
- Trade with a colony
- Fund an award
- Claim a milestone
- Convert plants / Convert heat
- Use a standard project
- Use an active blue-card action
- Use a CEO one-per-game action
- Send a delegate (Turmoil)
- Pass
- Sell patents
- End turn (intermediate)

Each one of these gets a **dedicated, persistently-visible, styled button** somewhere on the existing UI (in the resource cluster, in the top-bar overlays, on the colony tiles, etc.). The button's enabled state mirrors the option's presence in the action OR — same contract as the milestone/award/convert pattern above. When every action in this list has its dedicated button, the radio-UI rendering will be hidden via stylesheet flip; the underlying `WaitingFor.vue` / `OrOptions.vue` keep working unchanged for the (rare) cases of nested action menus during initial-action prompts.

**B. Mandatory sub-prompts** — things the server requires before the player can do anything else: payment selection, "pick a card to discard", "choose a player to steal from", "choose a space to place a tile", color picks for global events, payment of Reds tax, etc. These don't belong in a sidebar list — they're MODAL, not optional, and they arrive in response to an earlier action (or a forced game event).

For these, the fork uses a **centered modal popup** that:
- Renders over a darkened backdrop covering the entire viewport
- Disables interaction with the rest of the UI (no clicking the board, no opening overlays, no switching displayed player)
- Cannot be dismissed by clicking outside — only by completing the prompt (or by Reset/Undo flowing through the same submission infrastructure)
- Is sci-fi-styled to match the rest of the fork (dark glass + cyan accents + L-corner ticks)
- Hosts the actual input component via `<PlayerInputFactory>` so all existing input types work without per-component rewrites

The submission infrastructure (`WaitingFor.onsave()` → POST `/api/player-input`) stays unchanged — the modal just hosts the input UI in a different place visually. When the player completes the prompt and the server responds with a new `playerView` (with `waitingFor` cleared or pointing at the next sub-prompt), the modal unmounts naturally via `v-if`.

**The first pilot of this pattern is `SelectPaymentV2` invoked via `SelectPaymentDeferred`** (Fund Award as Helion, Reds tax, party actions, card payment actions, etc.). The second pilot is the **World Government Terraforming** prompt (`OrOptions` titled `'Select action for World Government Terraforming'`). The implementation lives in `WaitingFor.vue` (top-level detection) + `MandatoryInputModal.vue` (backdrop + centered card chrome) + the hosted input component itself.

**Card selection (draft / buy cards / research)** rides the same rails. `'card'` is in `MODAL_INPUT_TYPES`, so any top-level SelectCard prompt goes through the modal hosted by `CardSelectionContent.vue`. The same component handles draft picks (`min > 0`, mandatory) and buy-card prompts (`min === 0`, optional, with per-card cost). Mode is detected via the prompt title — server's `ChooseCards.ts` always builds "Select … to buy …" for research, which triggers the cost badge + "КУПИТЬ" button label; everything else falls back to "ВЫБРАТЬ". Cost-per-card reads `cards[0].calculatedCost` (server bakes it into every CardModel via `cardsToModel({played: false})`); MC check is `playerView.thisPlayer.megacredits`. PaymentV2 after buy is NOT this modal's problem — it's a separate `SelectPaymentDeferred` the server queues after we submit `{type: 'card', cards: [...]}`, and the existing `'payment'` modal route handles it. Single-click toggles selection in local state; double-click opens our own CardZoomModal with an `#actions` slot button that mirrors the toggle (handles capture-phase dblclick to suppress Card.vue's built-in zoom). Confirm is the only path to `onsave`; minimize-to-pill is inherited from MandatoryInputModal so the player can briefly inspect the board while picks stay intact.

**Adaptive layout (`CardSelectionContent.fit`).** The grid is no longer a static `nowrap` row + a 4-step viewport-only `zoom` media-query ladder (which made many cards overflow / become tiny regardless of count). It now WRAPS (`.card-selection__cards { flex-wrap: wrap }`) and a small JS fit engine — mirroring `HandCardsOverlay`/`PlayedCardsOverlay` — sizes it to the card COUNT + viewport: it sets `--cs-zoom` (continuous card scale) and `--cs-content-width` (the modal hugs a snug, centred grid). **The column/zoom/content-width math is a PURE function `cardSelectionRowPlan` in `src/client/components/cardSelectionFit.ts`** (extracted so it's unit-testable without a DOM — `fit()` itself is a no-op under JSDOM since rects are 0; covered by `tests/client/components/cardSelectionFit.spec.ts`). `baseZoom(n)` picks a starting scale (few cards big → `1.18`, many → `0.74`); `fit()` measures one slot's natural width and feeds it to the plan, which derives a **balanced** column count that fits the viewport width (`cols = floor(availW / slot)`, then rebalanced via `rows = ceil(n/cols); cols = ceil(n/rows)` so rows are even, never 5+1), sets a matching content width, then `fit()` shrinks `--cs-zoom` (sqrt steps, floor `0.5`, re-planning each step) ONLY if the rows would overflow `~0.92×innerHeight` — vertical scroll is the genuine last resort. **`padX` (the inset between `--cs-content-width` and the space available to slots) MUST count BOTH the `.card-selection` root padding AND the grid's OWN `.card-selection__cards` left/right padding (4px each side)** — forgetting the grid's 8px made every row overflow by ~8px and flex-wrap dropped the last card to a 2nd row. **Small picks (`n <= FIT_SINGLE_ROW_MAX` = 4 — the between-generation draft is 4→3→2→1, and the 4-card buy after it) are a HARD single row, enforced in CSS, NOT by JS tuning.** The grid gets `flex-wrap: nowrap` via the `.card-selection__cards--single-row` modifier (toggled by the `singleRowLayout` computed = `visibleCardCount <= FIT_SINGLE_ROW_MAX`), so a second row is impossible regardless of fit() timing (first paint, hard reload) or sub-pixel rounding — the earlier "tune the width to just fit" approach kept flashing / sticking on 3+1 because it relied on the JS measure landing in time and exact. With `nowrap` the row is a CSS invariant; the fit engine still runs to size it nicely (`cols = n`, the START zoom capped by WIDTH via `rowZoom = (availW − padX − (n−1)·gap − FIT_ROW_SLACK) / (n·naturalW)` so the row fits horizontally, `FIT_ROW_SLACK` = 12px breathing room), but it can no longer (mis)compute a wrap. Paired with the `max-content` fallback, a small pick is a correct centred single row even before / without JS. The height-fit loop can only shrink further, so the row stays intact; `FIT_MIN_ZOOM` (0.5) is the floor — only an extremely narrow viewport (where the slot's action-button min-width floor `FIT_SLOT_MIN_W` no longer fits n-across) falls back to wrapping. Width is capped to `FIT_MAX_CONTENT_W` (1560, < the 1640 modal max-width) so the content never bursts the modal frame. **The pre-engine FALLBACK width is `max-content`, NOT `100%`** (`.card-selection { width: var(--cs-content-width, max-content) }`). This matters because `.mandatory-input-modal__card` is a `width: auto` flex item in a centred flex wrapper — a `100%` child resolved against that auto-width parent COLLAPSED to a narrow box and the grid wrapped (the 3+1 bug seen on first paint / hard reload, before/if `fit()` runs). `max-content` makes the box hug the natural one-row width (clamped by `max-width: calc(100vw − 48px)`), so the FIRST frame is already a correct single row and the engine only refines zoom/centring. To kill the flash when the SAME component instance is reused across rounds (the draft's last 1-card round → a 4-card buy prompt, where the old narrow pinned width would briefly squeeze the new cards), the `playerinput` watch calls `resetFitVars()` (removes the inline `--cs-content-width`/`--cs-zoom` → back to the `max-content` fallback) BEFORE `deferFit`. `fit()` runs on `mounted` (+ a 240ms re-fit for art/font settle + a `document.fonts.ready` re-fit for hard reloads where fonts load late), on a `ResizeObserver`(root) + window `resize` via `scheduleFit` (rAF-coalesced), and on prompt / filter changes via `deferFit` (320ms, post-animation). When the first measure finds no laid-out card (`probeW <= 0` — JSDOM, or CSS/layout not ready post-reload) it RETRIES on the next frame, bounded by `FIT_MAX_RETRIES` (30 ≈ 0.5s), so the engine refines as soon as dimensions exist (the `max-content` fallback shows the right layout meanwhile). No-op under JSDOM (rects are 0 → keeps the CSS `--cs-zoom: 1` / `max-content` fallbacks). So: 1–4 cards → always a single centred row (zoom shrinks to fit width); 5–7 → a single row if it fits else a balanced grid; 8+ → a wider responsive grid. The media queries now only tweak gap/padding (NEVER `zoom` — that would fight the engine). Refs: `ref="root"` (the `.card-selection`) + `ref="grid"` (the `.card-selection__cards`).

### Client-side payment preview (Standard Projects)

The Standard Projects overlay (`StandardProjectsOverlay.vue` mounted from `PlayerHome`) is a hybrid case: the action OR is in `waitingFor`, but the player picks the project from the OVERLAY, not from the inline radio menu. Two payment paths:

1. **No alternative resources usable** — player has only M€ that can pay (or has alt resources but the project doesn't accept them). The handler in PlayerHome submits the full M€ payment directly through `WaitingFor.onsave()` with a nested `SelectProjectCardToPlayResponse`. No modal.
2. **Alternative resources usable** — at least one non-M€ resource the project accepts is owned by the player (heat for Helion / steel for City+Prefab / titanium for Luna TF / seeds for Greenery+Soylent / Aurorai data / Spire science). A client-side payment-preview modal opens — `StandardProjectPaymentContent.vue` hosted inside `MandatoryInputModal`. The PaymentFormV2 inside lets the player dial in the resource mix. **Two actions**: Confirm submits the chosen Payment; **Cancel closes the modal without any server round-trip** and restores the Standard Projects overlay.

The Cancel button is UNIQUE to this client-side flow. Regular server-driven modals (Fund Award, WGT, etc.) can't be cancelled because the server has already committed the preceding step. Standard Projects is different because nothing has been submitted yet — the player is constructing the response client-side. When extending this pattern to other "client-driven" flows (e.g. action menu reskin if it ever happens), follow the same shape: `MandatoryInputModal` wrapper + sci-fi content component + Confirm/Cancel button pair.

When extending the pattern to other input types (SelectCard for "discard X", SelectPlayer for "steal from", etc.), the contract is:

1. **Don't gate by `type` strings inline.** Detection lives in `WaitingFor.vue`. For inputs whose `type` itself implies "must be modal" (like `'payment'`), add the type to `MODAL_INPUT_TYPES`. For specific instances of a shared type (like the WGT `OrOptions` which shares `'or'` with the regular action menu), add the title to `MODAL_OR_TITLES`. The unified `shouldRouteToModal()` predicate gates both.
2. **The hosted component must work both inside and outside the modal.** When the radio-UI legacy flow renders the same input inline, it must still function. Don't add modal-specific props or state — render the same `<PlayerInputFactory>` either way.
3. **No backdrop-click dismiss.** The modal is mandatory by definition. If you need a "cancel and pick a different action" affordance, add an explicit Reset button inside the modal that calls the existing `WaitingFor.reset()` path.

### Modal "minimize" — let the player inspect the UI before deciding

Mandatory by definition doesn't have to mean "blocks every other pixel forever." Some decisions need context the player doesn't have at the moment the prompt fires — e.g. before funding an award the player may want to compare their tableau against opponents'; before raising a global parameter via WGT they may want to check whether an opponent is one step away from a milestone bonus. Forcing them to commit blind degrades the play experience.

To handle this, `MandatoryInputModal.vue` exposes a **minimize** affordance:

- A `↗` minimize button sits in the top-right of the modal card.
- Click it: the card collapses up + fades out, the backdrop becomes click-through, a sci-fi pill appears at the top of the viewport (below the top-bar buttons, NOT covering them) labelled `AWAITING DECISION — <prompt title>` with a pulsing cyan indicator dot.
- While minimized the player can interact with the whole rest of the UI (board, sidebar, top-bar overlays, opponent tableaus) but cannot trigger new actions — every action button reads `playerView.waitingFor` to decide enabled state, and because the server is still waiting on this modal prompt, all action buttons are naturally disabled. No extra client-side blocking required.
- The pill lives in its **own** `<Teleport to="body">` (separate from the modal wrapper) so it has an INDEPENDENT stacking context. z-index 107 — above top-bar buttons (z=105) and the Awards / Milestones badge strips that hang under them (z=106), BELOW bar-overlays (z=110). When the player opens an Awards / Milestones / Standard Projects overlay to inspect info before deciding, the overlay cleanly covers the pill; closing the overlay brings the pill back. The pill defaults to `top: calc(var(--bottom-bar-button-height) + 32px)` (≈ y=68px, leaves room above for the initial-draft pill-stack which sits at top:14 with height ~34px) and is draggable — see "Draggable pills" further down.
- Click the pill (or hit Enter/Space when focused): the modal expands back to full size with the original input state intact.
- Picker-mode (board-tile selection) is mutually exclusive with minimize — entering picker auto-cancels minimize, the modal hides via picker rules instead.
- A new top-level prompt firing (e.g. one SelectPayment resolves and a follow-up SelectPayment fires) auto-resets minimize state so the player doesn't miss the new prompt.

The minimize state lives in the `MandatoryInputModal` component itself, so it resets cleanly whenever the modal unmounts (e.g. server resolves the prompt).

### Modal "picker mode" — board interaction inside a modal prompt

Some modal-hosted prompts contain options that need the player to click on the board (e.g. the WGT `OrOptions` has an `'Add an ocean'` `SelectSpace` option). The modal's opaque backdrop would block board clicks, so we have a **picker-mode** mechanic:

- `MandatoryInputModal.vue` exposes a `mandatoryModalSetPickerMode(active: boolean)` function via Vue `provide()`. The key is exported as the `MANDATORY_MODAL_PICKER_SETTER` constant.
- `OrOptions.vue` `inject`s this function (`default: undefined` so it stays optional for inline use) and calls it from its `selectedOption` watcher: `true` when the picked option's `type === 'space'`, `false` otherwise. Also called with `false` in `beforeUnmount` so the flag doesn't get stuck.
- In picker mode the modal CSS class makes the backdrop transparent + `pointer-events: none` (board clickable underneath), and the card fades + shifts to the top of the viewport so it doesn't obscure the board. Hovering the card brings it back to full opacity so the player can re-pick a different option without finishing the board interaction.

If a future modal hosts a different input type that needs board interaction (e.g. a top-level `SelectSpace`), wire it to call the same picker-mode setter from its `mounted()` / `beforeUnmount()`. Don't reinvent the mechanism per input type.

- **Mandatory, top-level** (server-driven): `WaitingFor.vue` mounts it whenever `waitingfor.type === 'space'` — covers standard projects (city / greenery / aquifer placement after submit), card behavior tiles, global event placements. `cancellable: false`.

- **Mandatory, modal-nested** (modal-driven): `MandatoryInputModal.vue` mounts it whenever its picker-mode is active. The picker-mode setter now passes the option's `title` alongside the active flag so the banner reads "AWAITING PLACEMENT / Add an ocean" instead of falling back to a generic name. `cancellable: false` (the host OrOptions provides the "pick a different option" affordance via its faded-on-hover modal card; exposing Cancel here would duplicate it).

Three independent signal sources need coordination so the global lock (next paragraph) sees a single boolean. That's handled by `src/client/components/placementLockState.ts` — a module-level Vue `reactive()` object. The convert-plants and server-driven sources are read directly by PlayerHome's `placementPending` computed (from `playerView.waitingFor` and `convertPlantsPickerActive`); the modal-driven source is written to `placementLockState.modalPicker` by MandatoryInputModal's `provide()`'d setter and read by the same computed. When you add a NEW signal source for board-placement waiting (e.g. a future picker that doesn't go through any of these three paths), add a flag to `placementLockState`, write to it from the new source, and OR it into PlayerHome's `placementPending` computed.

**Draggable pills.** Both PlacementBanner and the MandatoryInputModal minimized pill default-position at `top: calc(var(--bottom-bar-button-height) + 32px)` (≈ y=68px) — leaves vertical breathing room above for the initial-draft pill-stack (`.initial-draft-pills`, top:14, h:34), and still keeps the pill clear of the topmost board labels. Drag clamps to the viewport so the pill can't be dragged off-screen. Click-vs-drag is distinguished by a 5-px movement threshold; a no-drag click still fires the host's @click handler (open details / restore modal), a drag-then-release suppresses the click via a one-shot capture-phase listener. When you add a new "awaiting prompt" pill to the same family, wire it through `makeDraggable` in the same `mounted()` / `beforeUnmount()` pattern and use the same `:style="dragStyle"` binding (returns `{}` at default position so CSS animations / transitions on `transform` are preserved).

The legacy `.wf-select-space` prompt header (containing the "Choose a location for ocean tile" text and the `<GoToMap>` "перейти к карте" link) is hidden via `display: none` in `placement_banner.less`. `SelectSpace.vue` still mounts through `PlayerInputFactory` so its `mounted()` hook attaches board-tile click handlers and adds `.board-space--available` highlighting — only the in-flow text header disappears.

When you add a new placement reason, add the enum value + label (server) and it flows through unchanged; the popover styling is automatic by reason `type`.

**Placement lock — turn-ending action buttons are disabled while the placement is pending.** `PlayerHome.vue` computes `placementPending` (true when either flow above is active) and a watcher toggles `body.placement-pending`. `placement_banner.less` then locks a curated list of action-button selectors. View interactions (opening Cards / Colonies / Awards / Milestones / Standard Projects overlays for inspection, browsing the hand, clicking opponent panels, reading the log, scrolling the board) stay fully interactive — only buttons that submit a turn-ending response are locked.

Lock mechanics (CSS for visuals, JS for actual blocking):
- CSS in `placement_banner.less`: each locked-selector match gets `opacity: 0.45 + filter: saturate(0.55) + cursor: not-allowed` for the visual "this is disabled right now" signal. **No `pointer-events: none`** — that would also suppress the native browser `title` tooltip we use for the explanation. **No `::after` overlay** — pseudo-elements don't intercept click events (click target is the host); the overlay-as-blocker trick doesn't work.
- JS click block in `PlayerHome.installPlacementGuards()`: a capture-phase listener on `document` runs BEFORE Vue's `@click` handlers (which compile to bubble-phase `addEventListener`). When the event target is inside a `PLACEMENT_LOCKED_SELECTORS` match, the guard calls `preventDefault` + `stopImmediatePropagation`. `stopImmediatePropagation` kills the event completely — no further listeners run, no submit happens. Capture phase guarantees the guard runs first regardless of where Vue attached the @click.
- Tooltip via the native browser `title` attribute. The same `installPlacementGuards` walks every currently-mounted locked button and sets `title="Сначала завершите текущее действие"` (translated via `translateText('Finish your current action first')`, i18n key already in `ru/ui.json`), caching any pre-existing `title` value into a `data-placement-orig-title` attribute. A capture-phase `mouseover` listener catches buttons that get mounted AFTER the lock kicks in (e.g. an overlay opening mid-placement) and applies the title on first hover. `uninstallPlacementGuards` restores originals on unlock.

**When you add a NEW dedicated action button** (per the "Action UI Rework" migration list above — sell patents, blue-card action, CEO one-per-game action, delegate placement, etc.), you MUST add its CSS class in TWO places:
1. The selector list in `src/styles/placement_banner.less` under `body.placement-pending { … }` (visual dim).
2. The `PLACEMENT_LOCKED_SELECTORS` array near the top of `src/client/components/PlayerHome.vue` (click block + tooltip).

Both must stay in sync — CSS alone won't block clicks, JS alone won't visually dim. Skipping either lets the player commit that action mid-placement, leaving the cancellable picker silently stale (cancellable case) or accepting an out-of-order submit that the server will partially process (mandatory case). Reasoning is the same as for the existing rule: while a tile placement is pending, the player must explicitly cancel via the banner before doing anything else that ends the turn.

## Journal (game log) — premium side-panel

The bottom-bar **Log** button no longer opens a `bar-overlay`; it toggles `journalState.open` (module-level reactive in `src/client/components/journal/journalState.ts`) and `src/client/components/journal/JournalPanel.vue` — a glass/HUD side panel pinned to the right gutter — is mounted at **App level** (`App.vue`, next to `DraftFlowOverlay`).

**Why App-level + module-state, not PlayerHome `data()`:** `App.vue` mounts `<player-home :key="playerkey">` and bumps `playerkey` on every server response — a full remount of the PlayerHome subtree. If the open flag lived in PlayerHome `data()` (or the panel were mounted inside PlayerHome), every board update would reset/destroy it and the journal would close itself + lose its generation/scroll. Mounting the panel at App level (it survives the remount, like the draft overlay) and keeping the flag in `journalState` makes it remount-proof: once opened it stays open until explicitly closed, preserving selected generation, scroll position and live-follow across updates. PlayerHome only READS `journalState.open` (the `journalOpen` computed → drives the `#player-home.journal-open` board-slide class + Log button active state) and WRITES it (`toggleJournal()`, which also closes any active `activeOverlay` so the journal isn't hidden behind one).

Opening it adds `#player-home.journal-open`, which slides the Mars board LEFT via `transform: translateX(...)` on `.player_home_block--board` (see `journal.less`). **The board scale is never touched** — only the centred box is translated, so `--board-scale` / `useBoardAutoScale` stay untouched (CLAUDE goal #3). The panel occupies the freed space between the board and the right sidebar; closing it slides the board back. The journal is NOT a `bar-overlay` (z-index 95). (`#player-home` creates no stacking context, so the App-level panel's z-index composes correctly with PlayerHome's overlays.)

**The journal has priority on screen space — non-modal overlays yield to it.** While the journal is open, the NON-MODAL overlays (`.bar-overlay`, `.top-bar-dropdown`, `.legacy-ui-overlay--open`) pull their `right` edge in to stop just left of the journal (`#player-home.journal-open <selector> { right: calc(var(--right-sidebar-width) + var(--journal-overlay-clearance)) }` in `journal.less`), so they shrink to the free area instead of covering it — same CSS-variable available-space mechanism as the board slide. `--journal-overlay-clearance = journal-gap + journal-width + 16px` breathing room; a `right` transition animates the adjust when the journal is toggled while an overlay is open. **Modals are the explicit exception** (`MandatoryInputModal`, `CardZoomModal`, the colonies overlay, draft overlay) — they are NOT in that selector list and may overlap the journal freely. When you add a new non-modal overlay that should yield to the journal, add its selector to those two rules in `journal.less`.

**Data source is reused, presentation is new.** The server side is unchanged: `GET /api/game/logs?id=&generation=` returns structured `LogMessage[]` (NOT HTML), each a template string + typed `LogMessageData[]` tokens (`LogMessageDataType`: PLAYER, CARD, CARDS, SPACE, COLONY, GLOBAL_EVENT, TILE_TYPE, AWARD, MILESTONE, PARTY, UNDERGROUND_TOKEN, …). `Log.parse()` splits a message into an ordered `(string | LogMessageData)[]`. The legacy `logpanel/` components (`LogPanel`, `LogMessageComponent`, `CardPanel`) are LEFT INTACT — still used by `SpectatorHome.vue` and `GameEnd.vue`. Do not delete them.

## Premium notification system (the live game-feedback layer)

The bottom-bar journal can be collapsed — so the fork has a **NotificationCenter** that surfaces important game events as floating sci-fi cards (top-right, where the journal opens) even when the journal is closed: opponents' card plays, your turn, mandatory decisions, milestones/awards, new generations, passes. **Architecture (mirrors the journal's module-state + App-level mounting):**
- `notificationTypes.ts` — the pure vocabulary: `NotificationKind` (`action-required` › `your-turn` › `warning` › `important` › `normal`, also the priority order), `NotificationModel` (render-agnostic: carries the journal `LogMessage` header + `JournalChildVM` children + headline pills + a CTA), `NOTIFICATION_TTL` (0 = persistent for turn cards), `MAX_VISIBLE_TRANSIENT` (3), `COALESCE_THRESHOLD` (3).
- `notificationModel.ts` — PURE mappers (unit-tested under the server runner, like `journalView.ts`): `diffRootNotifications` (diff the current generation's `buildJournalView` nodes against a seen `correlationId` set → new cards; SUPPRESSES the viewer's OWN ordinary actions, shows milestone/award highlights regardless of actor; returns ALL encountered ids so suppressed ones are still seeded), `buildEventChildren`-driven `summarizeImpact` (merged net pills + "+N details"), `coalesceBurst` (a >3 same-actor burst → ONE "Events: N" summary so an opponent's whole turn doesn't spam), `buildTurnNotification` (waitingFor → your-turn for the inline action menu [`ACTION_MENU_TITLES`, kept in sync with `WaitingFor.vue`], else action-required with the prompt text), `buildGenerationNotification` / `buildPassNotification`. **"Your turn" announces ONLY at the START of a fresh turn** — the `freshTurn` flag is a STRUCTURAL signal computed in the layer: the action menu titled `'Take your first action'` (NOT `'Take your next action'`, the continuation after a sub-prompt / a 2nd action) AND the viewer is NOT the lone non-passed player (`isLonePlayer()` via `game.passedPlayers`). It is NOT re-announced mid-turn (the menu returns as 'next action') nor for the lone player repeating turns. (An earlier transition-tracking impl — `viewerWasWaiting`, NOT-waiting→waiting — showed it only EVERY OTHER turn because the server doesn't always emit a waitingFor-cleared update between turns; the title+passed-players signal is robust to that.) action-required (sub-prompts) is unaffected — it always shows. **Both the pills (`chipRank`) AND the journal's own child rows (`buildEventChildren`'s final `childTier` sort) order GAINS first → the action's own result, then indirect gains, then discounts, then the cost/losses last** — "what I GOT before what I paid" (the colony test asserts this; gains-first applies to the JOURNAL too, not just the cards). **A still-visible root card is REFRESHED when its chain grows** — `recomputeRootImpact(events, correlationId, actor)` re-derives the pills + breakdown, and the layer's `refreshVisibleImpacts` updates header-bearing transient cards in place (no re-animate). This fixes a poll-timing race: an opponent's colony trade whose reward is a DEFERRED "add floaters to a card" pick (Titan, 2+ candidate cards → the trade PAUSES for the opponent's pick) could be diffed between the fee and the reward, so the first build saw only the spent energy — the refresh adds the gained floaters once they record. The CTA «В ЖУРНАЛ» (`labelKey: 'To journal'`, action `open-journal`) is a SECONDARY ghost button (`notification-card__cta--ghost`: no fill / no glow / smaller) — the card is the info, the journal is a peek; the primary CTAs (Посмотреть / Перейти) keep the bright plate.
- `notificationState.ts` — the module-level reactive store (survives the playerkey remount): a SINGLETON `turn` channel (your-turn/action-required) + a `transient` feed (cap 3 + `queue` overflow, promoted on dismiss) + the `seenRootIds` / `lastGeneration` / `passedSeen` / `seeded` lifecycle + a `settings` block (future on/off toggles). `dismiss` of a turn card records `dismissedTurnId` so an acknowledged prompt stays hidden until the prompt id CHANGES.
- `NotificationLayer.vue` — App-level (sibling of `<player-home>`, like the journal), Teleport to body. Fetches `/api/game/logs` + `/api/game/journal-events` for the current generation on every `playerView` change + a light 2.2 s poll (catches opponent actions during simultaneous research/draft, mirroring the journal). FIRST fetch seeds silently (no spam on load/reconnect). When the journal is OPEN, normal/important toasts are SUPPRESSED (the feed shows everything) but still marked seen; turn cards always show. END phase silences everything (the endgame experience owns the screen). Generation-went-backwards → `resetNotifications` (a different game opened in-session).
- `NotificationCard.vue` — compact (type label + glyph + headline + headline pills + CTA) → click (or the explicit «Подробнее» button — a chevron + accent count BADGE, unmistakably interactive) to EXPAND the breakdown, which REUSES `JournalChildRow` so the card and the journal never diverge. Lifetime indicator is a CSS-driven `scaleX` shrink (paused on `:hover` / `--expanded` via CSS — no per-frame JS), `animationend` → auto-dismiss. The generation shows as a CLEAR «Поколение N» line (`metaLine`), never a cryptic chip. **The YOUR-TURN card is persistent ONLY while the player is AFK: on the FIRST real activity (pointer move past a threshold / key / click / wheel — window listeners armed in `mounted`) it ARMS a ~5 s countdown** (`activityArmed` → `showProgress`/`effectiveTtl`), so it fades once the player is clearly playing but stays put if they stepped away. action-required stays persistent until resolved.

**Typed event-VARIANT styling — a player tells the TYPE at a glance, before reading.** `kind` (action-required/your-turn/warning/important/normal) drives BEHAVIOUR (priority / TTL / persistence / channel); a SEPARATE `NotificationVariant` (`notificationTypes.ts`) drives the VISUAL — accent colour + glyph + header label. `notificationModel.rootVariant(header, chain)` maps a journal root event → variant via the server-stamped `category` (+ chain event types): `card-play`→`play-card` (cyan ◈), `card-action`/`corporation-action`/`ceo-action`/`copied-action`→`blue-action` (azure ⟳), `standard-project`→`standard-project` (steel ⬡), `colony`→`colony` (violet ◉), `milestone`→`milestone` (GOLD 🏆), `award`→`award` (medal-purple 🏅), a standalone effect-trigger root→`passive-effect` (teal ✦), else `event`. Turn/generation/pass/warning builders set their own variant. To add a new variant: add the enum member, a `rootVariant` case, a `.notification-card--variant-X { --notif-accent }` rule + a glyph case.

**Hostile / NEGATIVE events — the viewer never silently loses something to another player.** A dedicated kind `negative` (priority just under your-turn, TTL 13 s, longer because a loss must not be missed) + the hostile variants `destroy` / `steal` / `production-reduction` / `production-transfer` (+ the Vermin `threat` / `vp-loss`). **100% client-derived from the VICTIM's `GameEvent`** — no server change for destroy/steal/transfer/reduction. `notificationModel.diffNegativeNotifications` scans the generation's events for ones where `e.player === viewerColor` AND there's a NEGATIVE impact AND `attackerOf(e)` (= `target.player` ?? a card/corp `source.owner` that ISN'T the viewer) — i.e. a cross-player attack, NOT a cost / own-spend / global event (those have no other-player attacker → skipped). It classifies from the single victim event: negative `impact.stock` + no `target.player` → **destroy**; + `target.player` → **steal** (the `stealing` flag set `target`); negative `impact.production` + no target → **production-reduction**; + target → **production-transfer** (shown as ONE linked event with a mirror `+X` gain chip for the attacker, never two independent rows). Grouped by `correlationId` (all the viewer's losses in one attacking action → one card), de-duped via a SEPARATE `notificationState.seenNegativeIds` (the loss lives inside the attacker's root action, which `diffRootNotifications` already saw). Only the **viewer-as-victim** gets a dedicated hostile card (the attacker/observer perspective is the ordinary root play-card card + its expanded breakdown — no spam); hostile cards show **even when the journal is open** (critical, like turn cards) and are **priority-aware** in `pushTransient` (a loss EVICTS a lower-priority visible card rather than queueing behind it). `NotificationCard` renders a hostile layout (`negative` meta): "Из-за <attacker chip> <source card chip>", a victim/attacker `−X → +X` flow for steal/transfer, a stock-vs-production marker («из запаса»/«доход»), and a `before → after` computed from the viewer's current `PublicPlayerModel` value (single-resource case). Styles + a calm ONE-SHOT impact glow + loss-pill flash in `notifications.less` (per-variant hostile accents: destroy red, reduction orange, steal purple, transfer cyan, vp-loss red-gold). **Vermin** is handled in TWO honest stages (its real rule: ≥10 animals on the card → −1 VP per city for EVERY player at SCORING, no mid-game VP events): the WARNING stage is the Vermin PLAY recognised by `CardName.VERMIN` on a `card-play` root → variant `threat` (amber, "potential VP loss"); the DAMAGE stage is a server signal — `Player.addResourceTo`'s Vermin hook, the moment `verminInEffect` flips false→true, roots a `'vp-pressure'` journal event (`category` added to `GameEvent.ts`) → client variant `vp-loss` (strong red-gold, shown to everyone incl. the owner). Forced-negative prompts (discard / pick-to-lose) already fire the persistent `action-required` card via the `waitingFor` path. Guarded by `notificationModel.spec.ts` (classification + Vermin variants) + `tests/events/verminJournal.spec.ts`. **Documented frontier:** observer-between-others hostile cards (folded into the root event today), the child-row-precise journal highlight (currently highlights the root group), and a hostile FLAVOUR on the forced-negative action-required card (needs a server "this prompt is hostile" marker).

## Played-cards board (the "РАЗЫГРАНО" overlay)

The bottom-bar **Played** button (`activeOverlay === 'played'`) opens `src/client/components/playedCards/PlayedCardsOverlay.vue` — a premium grouped, adaptive "project board" that replaced the legacy `bar-overlay--played` flat list. It shows the **currently-viewed** player's tableau (`displayedPlayer`, so switching the viewed seat re-points the board), grouped by card type, with a colour-accented player-context header.

It is driven by the same `activeOverlay` bar-overlay machinery: `toggleOverlay('played')` opens it, `handleOutsideOverlayClick` closes it on outside clicks (the overlay's own root `.played-board-overlay` is in that handler's exempt list, and an `dialog[open]` guard stops a click on the fullscreen card's backdrop from closing the board), and it's in `journal.less`'s overlay-yield rule so it shrinks left of the open journal. NOT in `PLACEMENT_LOCKED_SELECTORS` (it's a view overlay). It's a VIEW surface: it deliberately does NOT pass `actionUsed` to `<Card>`, so an active card whose action was already used this generation is **not dimmed** here — the dedicated Actions overlay owns action/used state.

**The board is a board-game-like TABLEAU with a HORIZONTAL SPLIT (`.played-tableau` is `flex-direction: row`), not a flat list.** The legacy CSS multi-column masonry (which wasted full-width rows on small groups, broke at 2-3 corps, and scrolled too early) is gone.
- **Identity RAIL** (`.played-tableau__identity`, LEFT): corporation / preludes / CEO ONLY — a compact setup zone stacked VERTICALLY (`flex: 0 0 auto`, `max-width: min(40%, ~2 cards × zoom)`, a right-edge divider). It does NOT hold project sections (an earlier "tuck small sections under identity" experiment was removed — the rail is identity-only). Identity cards use the **canonical fixed render** (`lightweight`). **ALL played cards take `--played-card-zoom` via the base `.played-card-item > .card-container` rule** (keying the zoom on `.played-group--project` only had once rendered non-project cards at natural size).
- **Main project band** (`.played-tableau__projects`, RIGHT, `flex: 1`): ALL project sections — Active / Automated / Events (`mainProjectGroups` = the non-identity visible groups) — as vertical columns, taking the whole right side at FULL height. Putting it BESIDE the rail (not below a full-width identity band) uses the top-right space AND makes the project SCALE depend only on the project area, not the identity render (fixes the "switching the viewed player makes cards tiny" bug). **Full cards by default** — see the planner.

Framework (`src/client/components/playedCards/`):
- `playedCardGroups.ts` — grouping + filtering + faceted chips. Six stable groups by `CardType` (corporation / preludes / CEO = `identity: true`; active / automated / events = project). Grouping is structural (`cardsOfTypes`, by `CardType`), never by text, and preserves **play order** (oldest→newest). **Two filter dimensions** (`PlayedFilter = {hiddenGroups, activeTags}`): `filterPlayedGroups` drops hidden-type groups + narrows surviving cards by tag; `buildPlayedTypeChips` (toggle-to-hide, count faceted by the tag filter) + `buildPlayedTagChips` (positive narrowing, faceted by the type filter, muted at 0 — mirrors the hand overlay) build the chips. `PLAYED_FILTERABLE_TAGS` = the same printed tag set the hand offers.
- `playedTableauFit.ts` — **PURE, unit-tested area-FILLING planner** (`planProjectBand`). Returns the card `zoom`, per-section `columns` + balanced `chunks`, `peekNatural`, `peek` (false → full cards), `density`. **FULL-CARD-FIRST:** it first runs the column-count SEARCH (`searchColumns`) in `'full'` mode (cards stacked FULL with a gap) — for each candidate column count `C`, `widthZoom` (falls with C) vs `heightZoom` (rises with C), take the min, the C with the LARGEST min wins (the cards as big as possible fitting BOTH dims, spread across the width). If that zoom is `>= fullMinZoom` the layout shows EVERY card full (no peek) — what the player asked for (fuller cards spread wide, not tall compressed piles). Only when there are too many cards for full to stay readable does it fall back to a `'peek'` search (overlapping cards fits more at a bigger zoom). `allocateColumns` hands columns to sections by crowdedness (Automated 40 wider than Active 21). For peek mode `peekNatural` fills the leftover height, clamped to `[minPeekNatural, cardNaturalH]` so the title ALWAYS shows. Covered by `tests/client/components/playedCards/playedTableauFit.spec.ts`.
- `PlayedCardsOverlay.vue` — root + the fit engine wiring. `railGroups` (identity) + `mainProjectGroups` (all project sections). `fit()` measures the body box, the REAL project-card natural height (`measureNaturalCardH` — `offsetHeight / zoom` of a `.played-group--project` card; CLAMPED to `[330, 470]` so a stray mid-transition measurement can't tank the layout), and the rail WIDTH (NOT height — the project height is the FULL band height, independent of identity), then calls `planProjectBand` for the project sections at `availW = totalW − railWidth` × full height and applies `zoom`, `peekNatural` (`--played-stack-peek-nat`; slot height = `peek-nat × zoom`), `bandSections`, a `density` class. `verifyShrink` is the measured safety net (shrinks ONLY the zoom, peek follows, down to `MIN_ZOOM`; only then does the body scroll). **`fit()` is SELF-CORRECTING** — after applying it re-measures the rail at the settled zoom and re-plans ONCE if the width shifted (>24px, e.g. on open / player switch), then flips `fitSettled` — so the reveal is FAST (a couple of frames), not a fixed wait; a separate non-gating deferred re-fit (280ms) catches late web-font / art shifts. **Reveal-when-settled (anti-jump):** `.played-tableau` is `opacity: 0` (layout kept so the engine can still MEASURE) until `ready` → fades in. `ready` flips true once `fitSettled` AND the shimmer-minimum has elapsed. `beginRevealCycle(force)` re-arms the hide on open / player / view switch (shimmer only for a big VISIBLE board, `visibleCardCount >= SHIMMER_THRESHOLD` = 50) AND on **every filter toggle** with `force = true` — which ALWAYS shows the premium `.played-loading` `Arranging cards` shimmer for `SHIMMER_MIN_MS` (300) regardless of card count (a filter re-plan otherwise reflows the cards visibly = a jump; the forced 300ms beat reveals the settled layout instead). The shimmer threshold uses the VISIBLE (filtered) count, so opening WITH filters on a small board can skip it. `scheduleFit()` (rAF-coalesced ResizeObserver / `@after-leave`) is the light refit. Also owns the shared `CardZoomModal` + Esc-to-close.
- `PlayedCardsFilters.vue` — type chips (toggle-to-hide, accent dot + faceted count) + tag chips (positive narrowing, tag icon + faceted count, muted at 0 with a `.premium-tooltip` hint). NOT native checkboxes.
- `PlayedCardsGroup.vue` — `variant='identity'` (idgrid, wrapping row of full cards) vs `variant='project'` (vertical columns per the overlay's `plan` + band-level `peek` flag). Project sections get `played-group--columns` (`flex: 0 0 auto` — rigid, so flex-wrap moves the WHOLE section to a new line rather than squashing the nowrap columns); identity sizes to content + wraps its cards. Cards split into the plan's balanced contiguous chunks (oldest-first; each pile reads top→bottom, newest at the bottom shown full).
- `PlayedCardsStack.vue` — one vertical pile; with `peek` every card except the last (newest) clips to `calc(var(--played-stack-peek-nat) × var(--played-card-zoom))` — `peek-nat` is the fit engine's natural-px peek (always ≥ the title height, so the NAME never clips, and it GROWS to fill spare height = fuller cards). **The clip is `clip-path: inset(-16px -16px 0 -16px)` (bottom-ONLY), NOT `overflow: hidden`** — the card's cost badge sits at `top:-10px` and the frame has an outer glow, both poking ABOVE the card box; `overflow: hidden` clipped them = the "top of the card looks cut off" defect. The negative top/side insets keep that chrome, the `0` bottom inset does the peek cut. Hover sets `clip-path: none` to reveal the full card. A non-peek pile shows every card full (a 16px gap). **Vertical, never horizontal.**
- `PlayedCardItem.vue` — wraps `<Card>` (`lightweight` always; the `autoTall` prop is unused now — identity + project both render at the canonical fixed height); `@click.capture.stop` opens fullscreen via the overlay's shared modal. No `actionUsed` (no dimming here).

## Effects overlay (the "ЭФФЕКТЫ" overlay — passive/ongoing effects)

The bottom-bar **ЭФФЕКТЫ** counter button (`activeOverlay === 'effects'`, between Played and Victory Points) opens `src/client/components/effects/EffectsOverlay.vue` — a premium MASTER-DETAIL panel of the displayed player's PASSIVE/ongoing effects (blue cards + corporations): a grid of effect blocks on the left, a per-effect "what it did this game" summary on the right. **100% client-derived, NO server change** (mirrors the colony-manifest precedent). The printed passive-effect GRAPHIC already lives in each card's `renderData`, which the client has via the static manifest (`getCard(name).metadata.renderData`) even though the live `CardModel` tableau doesn't carry it. So `src/client/components/effects/effectExtraction.ts` walks the displayed player's `tableau` (CardModel names + `isDisabled`), looks each card up, and filters its renderData to the passive-effect nodes:
- **What's an effect node:** `is:'effect'` at root level, OR the `is:'effect'` node(s) NESTED inside a `corp-box-effect` / `corp-box-effect-action` container (`is:'corp-box-action'` is skipped wholesale). Type guards from `@/common/cards/render/Types`.
- **effect vs action** (both render as `is:'effect'`!): excluded via `isPassiveEffectNode` — the DSL bakes an `'Effect: '` / `'Action: '` prefix into the node's description (last string of `rows[2]`), and the delimiter symbol is `:` (`CardRenderSymbolType.COLON`) for effects vs `->` (`ARROW`) for actions. Verified against all 995 cards: **77 in-scope cards → 83 effect blocks, 0 action leaks** (the desc-prefix check runs FIRST, so an `effect()` node that happens to contain an arrow SYMBOL in its content row — e.g. Supercapacitors' crossed energy→heat — is still classified as a passive effect).
- **Edge cases** (`EFFECT_OVERRIDES` map, `Partial<Record<CardName, {exclude?, renderWhole?, text?}>>`): a few cards encode the passive rule outside a clean `effect()` box. `renderWhole: true` renders the card's ENTIRE renderData root (the raw symbol rows ARE the effect graphic, via `CardRenderData`) + its description — used for `OLYMPUS_CONFERENCE` (drawn as raw root rows: science tag : science / OR / −science +card, no `effect()` node). `text:` shows a text-only block. `descFromMeta: true` appends `metadata.description` under a `renderWhole` graphic (for cards whose effect text isn't in the render — Olympus). `exclude:` drops a card. The other bespoke-render cards are now covered: `PROTECTED_HABITATS` + `NEPTUNIAN_POWER_CONSULTANTS` via `renderWhole` (the whole renderData IS the effect; Neptunian's effect text is already in its render so no `descFromMeta`). `SUPERCAPACITORS` needs NO override anymore — instead of a `text:` descriptor, its card was given a real `effect()` node (`eb.startEffect.energy(1).arrow(Size.MEDIUM, true).heat(1)` → energy → heat with a **crossed-out arrow** meaning "conversion is optional"; the immediate M€-production box stays a separate root row), so the generic scan extracts it like any normal effect card and the SAME graphic renders on the card AND in the overlay. The crossed arrow is a reusable render primitive: `ICardRenderSymbol.cancelled?: true` + `CardRenderSymbol.arrow(size, cancelled)` + the `b.arrow(size, cancelled)` builder overload; client-side `CardRenderSymbolComponent` adds `.card-symbol-cancelled`, and `cards_v2.less` draws a red diagonal line via `::after` (scoped under `.card-container`, so it works in the effect block too). Any future card can draw a struck-through arrow the same way. Per the brief, per-card `if`/override entries here are the SANCTIONED way to dab edge cases; the generic scan covers everything else. ## Actions overlay (the "ДЕЙСТВИЯ" overlay — activatable blue-card / corporation actions)

The bottom-bar **ДЕЙСТВИЯ** button (`activeOverlay === 'actions'`) opens `src/client/components/actions/ActionsOverlay.vue` — a premium **MASTER-DETAIL** surface (the interactive sibling of the Effects overlay). It REPLACED the legacy `bar-overlay--actions` flat `<Card>` list AND the earlier flat-grid-of-action-blocks-with-ВЫПОЛНИТЬ rework. **LEFT** = a compact 1–3-column grid of action GROUP cards (`ActionGroupCard.vue`: a SLIM header — name + type chip + stored-resource badge + group status — then one selectable `CompactActionCard.vue` ROW per render node, so a 2-action card is exactly the slim header + two compact rows tall, no per-row button). **RIGHT** = a static details panel (`ActionDetailsPanel.vue`) for the SELECTED action: source identity + ⤢ fullscreen, the FULL action text, the cost/result breakdown (`ActionResultsPreview.vue`), constraints (the can't-act reason), stored-resource count, "what happens next" (`ActionNextStepNotice.vue` variant `next` = «Далее»), and a **context-aware CTA** that OPENS the confirmation modal (never executes). Selecting a row in the master only FOCUSES the action (it never submits) — only the details CTA acts. Keyboard nav (Arrow keys move the selection across `flatRows`, Home/End, Enter/Space = CTA, Esc = close). Shows ALL actions — available / unavailable-with-reason / activated-this-generation — behind two faceted filters; nothing is silently hidden. Same premium dark-glass HUD frame + fullscreen. The two buttons are deliberately ADJACENT in the bottom-left rail (Cards · **Actions · Effects** · Played · VP) so the pair reads as passive (Effects) + activatable (Actions) abilities. (The old flat-block `ActionBlock.vue` is DELETED; the branch↔node helpers live in `actionBranchView.ts`.)

**The SELECTION is a RENDER NODE, the BRANCH is resolved from it — never a positional index.** Master rows are render nodes (`group.nodes`, manifest order); the preview's branches are in the server's BEHAVIOR order, and the two can DIFFER (Regolith Eaters / Atmo Collectors print "add" first but the server lists "spend" first). So `selectedKey`/`nodeIndex` identify a NODE, and both the details panel and the confirm modal resolve the matching preview branch via **`branchPositionForNode(group, branches, nodeIndex)`** (`actionBranchView.ts`) — which runs the SAME token-overlap match (`assignBranchNodes`) the modal graphic uses, then INVERTS it (which branch claimed this node). A naive node-i→branch-i map SWAPPED a multi-action card's «будет списано»/«вы получите» between its two actions — `branchPositionForNode` is the fix (guarded by `tests/client/components/actions/actionBranchView.spec.ts`). A combined-node card (1 node draws all branches, e.g. Self-Replicating Robots) resolves to `undefined` → the modal's in-modal branch picker. `PlayerHome`'s activate payload + `pendingCardAction` carry `nodeIndex` (not a branch position); the modal re-resolves from its own preview.

**Multi-action (`or`) rows read CLEAN — the leading "ИЛИ" is stripped, the alternation is a deliberate divider.** A multi-row group is ALWAYS an `or` action (a card has ONE action that branches). Each row's graphic has its leading OR connector stripped (`stripNodeOr`, `actionBranchView.ts`), and `ActionGroupCard` renders a slim `.action-group__or` "ИЛИ" badge BETWEEN consecutive rows — so the row graphic is never polluted by a stray "ИЛИ" symbol (the old leakage) while the choose-one semantics stay explicit.

**Selected state is on the ROW, not the group.** `.compact-action--selected` is the strong cyan ring; `.action-group--selected` is only a SUBTLE rim (the group merely CONTAINS the focused row) — so a 2-action card never reads as if BOTH actions are chosen.

**The master uses CSS MULTI-COLUMN packing, NOT a uniform grid** (`.actions-board__master { column-count: var(--actions-master-cols); column-fill: balance }` + `.action-group { break-inside: avoid; margin-bottom }`). A grid forces every row to the tallest card in it, so a 1-action card stranded beside a 2-action card kept a tall EMPTY gap below it — wasting space and forcing scroll even when the cards would otherwise fit. Multi-column packs each column independently and balances heights, so a 1-action card takes exactly one compact row's height and a 2-action card two — deterministic column-major order (NOT chaotic masonry), tight, scroll only when the content genuinely overflows. Because the layout is column-major, keyboard nav is Down/Up = ±1 (the next DOM item IS visually below) and Left/Right = ±`ceil(rows/cols)` (one column over).

**Double-click a row = quick-activate** (`CompactActionCard` `@dblclick` → `ActionGroupCard.activateRow` → overlay `@activate` → `PlayerHome.onActivateCardAction`). ONLY for an AVAILABLE action — `activateRow` no-ops otherwise, so a double-click on an unavailable action just leaves it selected with its reason shown, never opening a modal the player can't act on. Single-click still only selects. `onActivateCardAction` also guards on `placementPending` (the double-click is a separate path the `PLACEMENT_LOCKED_SELECTORS` CSS/JS guard — which only covers `.action-detail__cta` — can't catch).

## Fullscreen card viewer navigation (`CardZoomModal` — bounded card browser)

The fullscreen card viewer (`src/client/components/card/CardZoomModal.vue`) is OPT-IN navigable: when a host passes an ordered `cards` list (> 1) it becomes a bounded left/right browser (HUD arrows + `1 / N` counter + ← / → keys + a directional slide), letting the player decide IN fullscreen across the SAME visible list instead of close → pick → reopen. Omit `cards` (the default) and it stays the classic single-card preview — no arrows, no keys, identical to before. This is why **view-only / grouped surfaces stay single-card for free**: PlayedCards (`РАЗЫГРАНО` — grouping must not be flattened), Journal chip, `Card.vue`'s built-in zoom, `InitialDraftFinalConfirmContent`, `DraftedCardsPile`, `VictoryPointsOverlay` all pass only `:card`, so navigation never appears there.

The inner card markup + per-card computeds were extracted into `CardZoomCard.vue` so the visible card AND the off-screen preload neighbours (`currentIndex ± 1`, render/asset-warmed) reuse one component without duplication or a circular import (importing `Card.vue` would cycle — it hosts `CardZoomModal`).

**Contract for opting a context into navigation:**
1. Pass `:cards="<the visible list, in the SAME filtered/sorted order the player saw>"` (e.g. `sorted.map(e => e.card)` for the hand, `visibleCards` for selection). Navigation walks THAT order, not the raw collection.
2. Listen `@navigate="zoomCard = $event"` (first arg is the current `CardModel`). The viewer owns `currentIndex` internally and re-emits the current card on every step (and on `show()`); the host re-points its zoom variable so the `#actions` slot + `:selected` halo ALWAYS reflect the card on screen — the action button applies to the current card, availability/selected/reason all update live. The viewer keeps NO selection state of its own; it reads the host's shared module/data state (sale/select selections, `selected[]` arrays), so fullscreen and the source overlay never diverge.
3. A non-removing action (select/unselect/toggle) must NOT close the viewer — stay on the card so the player keeps browsing (`onFullscreenBuyToggle`, the initial-draft `onFullscreenToggle`s, hand `toggleSelectZoom` were all changed to stay open). A committing single-pick (draft `Select`, corp/CEO `Select`) still closes + submits. An action that REMOVES the current card from the list (drawn-cards `ВЗЯТЬ`) advances to the next item; the viewer's `cards`-prop watcher clamps `currentIndex` and plays a calm "consume" swap, or the host closes the viewer when the list empties.
4. `index` is an optional start-seed only (handy for duplicate-name lists); no bundled context binds it two-way — they all track the card via `@navigate`.

Internals worth knowing before extending: the slide is a single keyed `<transition>` (`:key="activeCard.name"`) with direction-aware names (`card-zoom-next/prev/consume`, `card-zoom-rm` under `prefers-reduced-motion`); the leaving card is pinned `position:absolute` so it overlaps rather than stacks; `slideDir` is set BEFORE `currentIndex` so the same render flush animates the right way. Rapid presses are blocked by an `animating` guard (released on `@after-enter`, fallback timer = `ANIM_MS + 80`) — `currentIndex` is the single source of truth so it can never race. Each new card is re-fit on `@enter` before paint (`fitCardToViewport(el)`); nav mode reserves extra chrome (~200px horizontal, ~64px top) so the card lands in a clean centre band with the controls fully clear of it. Arrows are bounded — at the first/last card the control is DISABLED-but-VISIBLE (dimmed, never hidden), so the edge of the set is felt. **A/D are deliberately NOT bound** — `HomeMixin.navigatePage` already owns `KeyA`/`KeyD` (jump to board / hand) on a window listener that ignores open dialogs, so binding them would double-fire; Arrow keys have no such global owner.

**Viewer layout = three separated layers (v41-b).** The fullscreen viewer is a column: a DEDICATED top zone (`.card-zoom-topbar`, in-flow — holds the premium `.card-zoom-counter` capsule) ABOVE the card, the card stage in the middle, the actions panel BELOW. The counter is its OWN viewer element, never on the card and never colliding with the on-card "ВЫБРАНА" ribbon (the three layers are counter→above, ВЫБРАНА→on-card, actions→below). The prev/next controls live in a `.card-zoom-midrow` flex row (`[ prev ] [ stage ] [ next ]`, `justify-content: space-between`) so they're vertically centred ON the card and pinned to the GUTTERS (stable positions that don't drift as cards of different widths page through) — clearly OUTSIDE the card, never grey buttons laid on it. The midrow is a plain centred flex row in the single-card (no-nav) path, so every existing single-card use is unchanged. Each control is a hit-area button hosting a clipped dark-glass `.card-zoom-nav__plate` (cyan inset rim + `filter: drop-shadow` glow — `box-shadow` would be clipped by the angular `clip-path`) + a chevron, with full default / hover (brighter rim + glow + directional nudge) / pressed (scale + deepen) / disabled (dimmed-but-present) states, in the same dark-glass + cyan-rim language as the `cab-*` action buttons. New nav styles live in `preferences.less` (`.card-zoom-topbar / -counter / -midrow / -stage / -nav(+__plate/__chevron) / -preload` + the transition classes); arrow aria-labels (`Previous card` / `Next card`) are in `ru/ui.json`.

## Card-action button design system (`card_action_buttons.less` — `cab-*`)

Every card-selection flow (hand overlay "КАРТЫ В РУКЕ", sale-patents mode, the fullscreen viewer, buy / draft / select / research modals, initial-draft steps, drawn-cards reveal) draws its buttons from ONE angular/HUD language so nothing reads as a premium HUD plate in one place and a plain rounded web-button in another. The system lives in `src/styles/card_action_buttons.less` (imported in `common.less` BEFORE every consumer so the mixins resolve). It is LESS-mixin-only — no markup changed; existing class names keep working, they just pull a shared look.

**Apply pattern:** `.cab-base()` on the BASE class (so its `&__label`, `&__glow`, `&::before` selectors resolve onto the real BEM children), then a `.cab-palette-*()` on the base AND/OR a `--modifier` (palette mixins only touch `&` / `&::before` / `&::after` / `&:hover`, never BEM children, so they're safe on a modifier). The canonical silhouette is `@cab-shape` (a px-based parallelogram → width-independent); the chassis is the parent `background` showing through a 1-px-inset `::before` (rim brighter than body).

**Roles (visual weight low → high) + where used:**
- `cab-palette-secondary` — ЗАКРЫТЬ / ОТМЕНА (quiet dark glass). `hand-sale-cancel-btn`, modal `--secondary`.
- `cab-palette-neutral` — ВЫБРАТЬ (steel/cyan dark glass). `hand-card-sell-btn` rest, `card-selection__card-action-btn`, `initial-draft-pick__card-btn`, fullscreen `hand-zoom-sell`.
- `cab-palette-deselect` — СНЯТЬ ВЫБОР (dark glass + **amber rim**, NO breathing). **The key tone-down:** it used to be a solid bright fill that competed with the CTA; it is now deliberately calmer than any CTA, everywhere (`--selected` / `--unselect` in hand, card_selection, initial_draft, modal). ## Modern modal inputs (card-play sub-prompts)

When a played card (or any forced event) needs an answer, the server returns a `PlayerInput` as the TOP-LEVEL `waitingFor`. Those sub-prompts used to fall through to the legacy radio stack inside the hidden `.legacy-ui-overlay` (invisible unless the player manually opened Legacy UI). They now route to premium surfaces. `WaitingFor.shouldRouteToModal()` decides:

- The per-turn **action menu** (`or` titled `'Take your first action'` / `'Take your next action'`) is the ONE top-level `or` that stays inline (driven by the fork's dedicated action buttons). Every OTHER top-level `or` (card-driven OrOptions, WGT) routes to the modal.
- `MODAL_INPUT_TYPES` = `payment, option, player, amount, resource, resources, productionToLose, and, projectCard` — these can only ever be a mandatory sub-decision, so every instance routes to `MandatoryInputModal`. EXCEPTION for `projectCard`: `WaitingFor.useModalForCurrentInput` suppresses the modal when the top-level `projectCard` is routed to a dedicated overlay — `handPlayPrompt(view)` ("play a card FROM HAND" — EccentricSponsor / EcologyExperts via `PlayProjectCard`, candidates ⊆ hand) → КАРТЫ В РУКЕ overlay's PLAY mode; `standardProjectPlayPrompt(view)` ("play a standard project" — EstablishedMethods via `SelectStandardProjectToPlay`, candidates are standard projects NOT in hand) → Standard Projects overlay. BOTH reuse their overlay's exact existing pick → pay → submit flow; the only difference is the EMPTY response path (`playProjectCardAction` / `standardProjectsAction` return `{path: [], input}` for the top-level prompt, and `submitPlayCard` / `submitStandardProjectPayment` submit the BARE `{type:'projectCard', card, payment}`). The modal stays as a fallback only for a degenerate `projectCard` matching neither. Both overlays auto-open + minimize to the SHARED mandatory pill (`handPillVisible`/`handPillLabel`/`handPillTitle`, restore re-opens the owning overlay). The action menu's own `projectCard` is a NESTED option inside the top-level `or`, never top-level, so it's unaffected.
- Handled by dedicated surfaces OUTSIDE the modal (NOT routed here): `card` → `DraftFlowOverlay` / hand-select overlay; `space` → `PlacementBanner`; `colony` → `ColoniesOverlay`; `initialCards` → `InitialDraftFlowOverlay`; a FREE **award-funding** OrOptions → the modern **AwardsOverlay** in free-sponsorship mode (see below).

### Contextual choice modal (triggered-effect / on-play / deferred `OrOptions`)

A top-level `OrOptions` from a TRIGGERED effect (Pharmacy Union's science-tag "turn face down / do nothing", Olympus, Splice, Recyclon, Viral Enhancers, Mars University, Neptunian Power Consultants), an on-play decision, or a deferred action used to render via the bare `ModernOptionPicker` — a context-less "ВЫБЕРИТЕ ВАРИАНТ" list: no source card, no "why did this appear", weak result preview, no downside. The fix is the premium **`ContextualChoiceContent.vue`** (`src/client/components/modalInputs/`): a 2-column composition mirroring `CardActionConfirmContent` — the **source as a premium framed ARTIFACT on the LEFT** (`.contextual-choice__src-frame` dark-glass container + type-tinted glow rim + corner tick + hover zoom-hint; the card `:key`-ed, `autoTall`, click → fullscreen), a **kind chip + source name + framed "event summary" TRIGGER block** on the RIGHT, then the options below. The whole decision (source + summary + options) lives in ONE `.contextual-choice__scroll` area that scrolls AS A UNIT when it overflows — so a TALL card (corporations) is NEVER clipped inside a small fixed box (the earlier defect). It does NOT re-implement option handling: it WRAPS the (header-suppressed) shared `ModernOptionPicker` (`hideHeader` prop), so every interaction (nested inputs, board picker, hand pick, skip-split, disabled targets) is reused.

**Interaction model = ONE-CLICK, with inline-confirm only for RISKY options.** `ContextualChoiceContent` passes `:controlled="true"` (a SAFE leaf option commits on the FIRST click — no redundant footer confirm button; "Ничего не делать" / a plain gain are one-click) **plus** `:confirmRisky="true"`. In `confirmRisky` mode an option the server flagged RISKY — it carries a non-numeric `metadata.tradeoff` (irreversible side-effect, e.g. Pharmacy Union's "card turned face down") OR `warnings` — does NOT commit on the first click: it ARMS an inline **confirm DRAWER** attached under the SAME card (`.modal-input__option-card--confirming` squares the bottom + an amber `.modal-input__option-confirm` drawer with real Cancel/Confirm buttons — a SIBLING of the card, never nested, so it's valid HTML + keyboard-accessible; Esc disarms). A second click on the armed card also confirms. Both `controlled` + `confirmRisky` are OPT-IN props (default false) so every OTHER `ModernOptionPicker` consumer (WGT top-level select→confirm, the action/play-modal hosted steps) is byte-for-byte unchanged. The primary options are wrapped in a transparent `.modal-input__option-slot` flex item (so the drawer has a home); skip options stay in their own neutral block.

**Server marker (the only server change is additive + backward-compatible).** `BaseInputModel.choiceContext?: ChoiceContext` (`{source: {kind: 'card'|'corporation'|'standardProject'|'colony'|'system', card?}, trigger?, mode?: 'optional-effect'|'effect-choice'|'attack'|'reward'}`) — `src/common/models/PlayerInputModel.ts`. Set CO-LOCATED in the card via `OrOptions(...).markChoiceContext(...)` (`BasePlayerInput.markChoiceContext`), serialized CENTRALLY in `Server.getWaitingFor` (next to `startGamePrompt`/`awardFundingPrompt`). Builders are co-located-friendly: `src/server/inputs/choiceContext.ts` (`cardEffect(card, trigger, mode)` derives corp-vs-card from `card.type`; `attackEffect`/`effectChoice`/`systemChoice`/`namedCardEffect`). Per-option RESULT chips reuse `ActionEffect`: `OptionMetadata` gained `effects?` (rendered by the shared `ActionEffectChip`), `tradeoff?` (a NON-numeric downside shown as an amber warning — Pharmacy Union's "card turned face down"), and `description?`; chip builders in `src/server/inputs/optionMetadata.ts` (`chip`/`trChip`/`optionResult`). `ModernOptionPicker` renders all three on every option card (contextual or not).

**Routing:** `ModalInputHost.premiumComponent` returns `ContextualChoiceContent` for `type==='or' && choiceContext !== undefined`, else `ModernOptionPicker` (unchanged). A prompt WITHOUT the marker is byte-for-byte unaffected. Styles: `src/styles/contextual_choice.less` (+ `.modal-input__option-effects/-tradeoff/-desc` in `modal_inputs.less`). i18n: `ru/ui.json` (kicker/kind labels + trigger sentences). Playground: `?modalPlayground` has two `contextual`-tagged scenarios. Tests: `tests/models/choiceContext.spec.ts`, `tests/client/components/modalInputs/ContextualChoiceContent.spec.ts` + `ModalInputHost.spec.ts`.

**NOT a gap (already premium elsewhere):** blue-card `action()` branch picks render inside `CardActionConfirmContent` (the ДЕЙСТВИЯ overlay); on-play bespoke choices with a `cardPlayPreview` hook are pre-collected in the play modal. **Documented phase-2:** the shared deferred-action attack prompts (`StealResources`/`RemoveAnyPlants`/…) already have rich per-target metadata + disabled reasons; they only lack the source-card preview (needs threading the source `ICard` through the Executor). Full status: `CHOICE_CONTEXT_AUDIT.md`. **To add a new contextual choice:** `.markChoiceContext(cardEffect(this, '<trigger i18n key>', '<mode>'))` on the `OrOptions` + per-option `.withMetadata(optionResult({effects, tradeoff}))`; add the trigger key to `ru/ui.json`.

### Option UI metadata (rich choice cards)

A generic `OrOptions` only carries text labels, so the client can't render an icon / player chip / impact preview without semantic context. The OPTIONAL `OptionMetadata` (`src/common/models/PlayerInputModel.ts`, on `SelectOptionModel.metadata`) carries that context — `{kind, icon, amount, player:{color,current,resulting}, global:{current,resulting,unit}}` — **alongside** the text, never replacing it. Backward-compatible: an option with NO metadata renders via the text fallback (the previous behaviour). Server side, `SelectOption.withMetadata(...)` attaches it (serialized in `toModel()`), and `src/server/inputs/optionMetadata.ts` has factory helpers so cards declare intent without boilerplate — add a factory per recurring pattern rather than hand-building metadata at the call site. Current helpers:

- `removeResourceFromPlayer(target, resource, amount, current)` / `stealResourceFromPlayer(...)` — destructive player-target (kind `resourceRemoval` / `steal`), with a `current → resulting` preview on the target.
- `gainResource(resource, amount)` — standard-resource gain (kind `resourceGain`).
- `globalParameter(icon, steps, current, resulting, unit?)` — raise a global parameter (kind `globalParameter`); `icon` ∈ `temperature` / `venus` / `oxygen` / `ocean`, `unit` is the symbol (`°C` / `%`), preview reads `global.current → global.resulting`.
- `addResourceToCard(cardResource, amount?)` / `removeResourceFromCard(cardResource, amount?)` — card-resource add/remove (icon = the `card-resource-<key>` suffix, derived from the `CardResource` value by lowercasing + spaces→hyphens; correct for every in-scope resource, only the out-of-scope fan card `RESOURCE_CUBE` differs).
- `skip()` — a "do nothing / skip" option (kind `skip`).

**Icon families are resolved in one shared place: `src/client/components/modalInputs/optionIcons.ts` `iconClassFor(icon)`.** It maps an icon-key to one of three CSS class families — standard resources → `resource_icon resource_icon--<key>`, global parameters → `wgt-icon wgt-icon--<key>`, everything else → `card-resource card-resource-<key>`. `ModernOptionPicker` (lead icon + preview icon) and `ModernAmountSelector` both call it, and `modal_inputs.less` normalises the size of all three families via the shared `.modal-input__option-icon` class (the `wgt-icon` set is otherwise only sized inside a compound selector, so the standalone normalisation is required). When you introduce a new icon-key, put it in the right set in `optionIcons.ts` — don't hand-build the class string at the call site.

Client side, `ModernOptionPicker` reads `optionMeta(opt)` and renders the rich card (player colour dot + name, the resolved icon, `current → resulting` preview when either `player` or `global` carries both values); `optionColor` prefers `metadata.player.color` and falls back to a PLAYER token in the title Message; `optionActionText` uses the `buttonLabel` verb for player-target options (player + amount already live in the chip + preview) and the descriptive title otherwise.

**`SelectAmount` carries the same icon vocabulary.** `SelectAmountModel` has optional `icon?` / `unit?`; the server `SelectAmount` constructor takes an optional 6th `options?: {icon?, unit?}` (backward-compatible — omit for a bare stepper). `ModernAmountSelector` renders the icon + unit beside the value via the same `iconClassFor`.

**`SelectPlayer` is a PREMIUM TARGET PICKER, not a name list (`ModernPlayerPicker`).** Each candidate is a rich glass *target card*: colour accent bar + dot, name, corporation (looked up client-side via `getCard(card.name).type === CardType.CORPORATION` over the public `tableau`), and — when the server supplies `{icon, amount, scope}` — a **per-target impact mini-HUD** (`current → resulting`, with a "ПРОИЗВОДСТВО" label over the framed icon for production). The header is a short heading + lighter subtitle (the full server instruction) + a secondary HUD effect chip; the flow is **select → confirm** (click selects, a premium CTA commits) so a destructive pick is never a single mis-click. `SelectPlayerModel` carries optional `icon?` / `amount?` / `scope?: 'stock' | 'production'`; the server `SelectPlayer` constructor takes an optional 4th `options?: {icon?, amount?, scope?, disabled?}`. **`scope` is the production-vs-stock distinction** — `'production'` reads each player's `*Production` field and draws the icon inside a brown "production frame" (`.modal-input__prod-frame`), `'stock'` (default) reads the plain resource field. Per-target current/resulting are computed **client-side** from the public player models (every player's stock + production is public), so the server only sends the hint — no per-target payload.

**State semantics are kept visually distinct** (CLAUDE goal: never make a player guess): SELECTED = the one strong cyan accent; SELF = a *local* amber "Это вы" chip ONLY (never a whole-card highlight, so self ≠ selected); DISABLED = subdued + non-interactive but still shows its reason; muted = a `resulting === current` no-op (soft, still clickable).

**Unavailable targets are SHOWN, not hidden.** Instead of silently dropping a relevant-but-unpickable player, the server passes them via `SelectPlayer`'s `disabled: ReadonlyArray<{player, reason}>` → `SelectPlayerModel.disabledPlayers` → greyed, non-selectable cards with the reason. `DecreaseAnyProduction` surfaces opponents who can't be reduced (`Production already at minimum` / `Production is protected`, mirroring `Player.canHaveProductionReduced`); `CometForVenus` surfaces opponents with `No Venus tag`. The selectable `players` list (what the server validates against) is unchanged — disabled entries never submit. Reasons are English i18n keys translated client-side (`ru/ui.json`). Wired (impact): `DecreaseAnyProduction` (shared → Energy Tapping & every production-attack card, `scope: 'production'`), `Flooding` / `CometForVenus` / `LawSuit` (`scope: 'stock'`, M€). A bare `SelectPlayer` with no icon (e.g. Colony "discard a card") still renders rich name+corp cards.

**`OrOptions` carries the same disabled-target idea via `OrOptionsModel.disabledOptions` (`DisabledOptionModel[]`).** The server `OrOptions.setDisabledOptions(...)` attaches informational, non-selectable entries; `ModernOptionPicker` renders them under an "НЕДОСТУПНЫЕ ЦЕЛИ" heading as greyed twin cards (player chip + greyed icon + reason). Built via the `disabledPlayerTarget(target, icon, reason)` helper in `optionMetadata.ts`. Wired: `RemoveAnyPlants` (`No plants to remove` / `Plants are protected`), `StealResources` (`Nothing to steal` / `Resources are protected`), `Sabotage` + `HiredRaiders` (`No resources to remove` / `Resources are protected`). Never processed — `disabledOptions` is a separate field, so option indices (and `OrOptions.process`) are unchanged.

**Colony TRADE flow is a confirm-first premium modal (`ColonyTradePaymentModal`).** Pressing **ТОРГОВАТЬ** on a colony NEVER trades instantly — `PlayerHome.onColonySelected` (trade mode) ALWAYS opens `ColonyTradePaymentModal` (even with one pay path), and the server action fires only on the modal's **ПОДТВЕРДИТЬ ТОРГОВЛЮ** (select → confirm). The colony-tile action button is mode-aware: `ColonyTile`'s `mode` prop → `selectLabel` renders `Trade` in trade mode, `Select` for SelectColony. The modal shows a colony summary (planet + name + the reward at the current track position via `BenefitGlyph`/`tradeAtIndex`) and every payment path as a rich option card (resource icon + cost + `In stock: current → resulting`). UNAFFORDABLE standard resources are shown DISABLED with a reason instead of vanishing: server `Colonies.tradeWithColony` adds them to the inner `OrOptions.setDisabledOptions(...)` (via each `IColonyTrader`'s new optional `optionMetadata()` + `disabledReason()`), and the bespoke modal renders `payOr.disabledOptions` greyed (`data-hint` tooltip). The self-resource preview rides a NEW `OptionMetadata.resource: {current, resulting}` field. The trade-payment OrOptions is the `'Pay trade fee'` inner OrOptions of the `'Trade with a colony tile'` AndOptions — `PlayerHome.findTradeColonyContext` extracts both `paymentOptions` and `disabledPayments`. (Localization gap fixed: `'Pay ${0} energy'` had no ru key — the energy option rendered untranslated "PAY 3 ENERGY"; added to `ru/ui.json`.) The bespoke modal is in `?modalPlayground` (a dedicated section, since it isn't routed via `modal-input-host`) so colony modals stay in the audit gate.

**Trade-bonus transparency block.** When colonies are already built on the tile, trading hands every colony owner the FIXED `metadata.colony` bonus (a SEPARATE, track-independent reward — Luna 2 M€, Ceres 2 steel — NOT the trade reward). `ColonyTradePaymentModal` surfaces this as a SECONDARY block under the colony summary (`.colony-trade-pay__bonus`, quieter than the summary so it never steals focus from the payment decision): the bonus benefit once via `BenefitGlyph` (`colonyBonus` computed = `getColony(name).colony`, built like `tradeReward`) + a player chip per owner with an explicit `×N` when one player holds several colonies here. **`SelectCard` carries it via `SelectCardModel.disabledCards` (a SEPARATE list of `CardModel`s, NOT mixed into `cards`).** The server `SelectCard` constructor takes `disabled?: ReadonlyArray<{card, reason}>` in its config; `toModel` serialises them through `cardsToModel(…, {enabled: all-false, disabledReasons})` so each disabled `CardModel` gets `isDisabled = true` + `disabledReason` (new `CardModel` field). Keeping them OUT of `cards` is deliberate: `cards` stays the selectable set, so `SelectCard.process` (which validates membership in `this.cards`) rejects them for free, auto-select/count logic is unchanged, and existing tests that assert `cards.length` don't break. `CardSelectionContent.vue` merges the two for display behind an **All / Available / Unavailable filter** (default **Available**, only shown when `disabledCards` is non-empty); a disabled card renders greyed with a muted reason chip (`data-hint` premium tooltip for the full text) instead of an action button. **Scope is case-specific, not a count heuristic** — the deferred decides what's "relevant". `RemoveResourcesFromCard.getUnavailableTargetCards` enumerates the SAME resource-type cards (`getResourceCards(resourceType)`, regardless of count) that aren't selectable and tags each `No resources on this card` (count 0) or `Card resources are protected`. ### Metadata contract — CHECKLIST for adapting / adding cards

**This is the single source of truth for what UI metadata a card's prompts must carry.** Whenever you adapt a blue-card action, an expansion card, or a card pulled from upstream `master`, run its prompts through this checklist so the premium modals render rich instead of falling back to bare text. **Everything here is OPTIONAL + backward-compatible** — a prompt with no metadata still works, it just looks plain. **When you EXTEND the framework (new field, helper, icon family, or disabled support for a new input type), update THIS checklist, the `OptionMetadata` type / model, and add a `?modalPlayground` scenario — in the same change.**

Identify what prompt the card produces, then attach the matching metadata:

| Prompt the card builds | What to attach | Helper (`src/server/inputs/optionMetadata.ts`) / API | Reference card |
| --- | --- | --- | --- |
| `OrOptions` of `SelectOption`s, each removing/stealing from a chosen player | `.withMetadata(...)` on every option **+ explicit verb `buttonLabel`** (NOT default `'Confirm'`) **+ `.withMetadata(skip())` on the do-nothing option** | `removeResourceFromPlayer(target, resource, amount, current)` / `stealResourceFromPlayer(...)` | RemoveAnyPlants, Sabotage, HiredRaiders |
| `SelectPlayer` where the effect is constant across candidates | 4th arg `options: {icon, amount, scope}` (`scope: 'production'` for production decrease, `'stock'` for stock removal) | `SelectPlayer(players, title, buttonLabel, {icon, amount, scope})` | DecreaseAnyProduction, Flooding, CometForVenus, LawSuit |
| `OrOptions` raising a global parameter | `.withMetadata(globalParameter(icon, steps, current, resulting, unit))` (`icon` ∈ `temperature`/`venus`/`oxygen`/`ocean`; `unit` `'°C'`/`'%'`) | `globalParameter(...)` | Atmoscoop |
| `SelectAmount` spending/converting N of a resource | 6th arg `options: {icon, unit?}` | `new SelectAmount(title, label, min, max, maxByDefault, {icon})` | Insulation |
| `SelectOption` adding/removing a **card** resource | `.withMetadata(addResourceToCard(cardResource))` / `removeResourceFromCard(cardResource)` | `addResourceToCard(...)` / `removeResourceFromCard(...)` | OlympusConference |
| `SelectOption` gaining a standard resource | `.withMetadata(gainResource(resource, amount))` | `gainResource(...)` | (factory ready, wire as needed) |
| Any "do nothing / skip / cancel" `SelectOption` | `.withMetadata(skip())` | `skip()` | every skip option above |

**Always also surface UNAVAILABLE targets (don't let the server silently drop them):**
- `OrOptions` attack/steal: build the excluded opponents and `.setDisabledOptions([disabledPlayerTarget(target, icon, reason), …])` on the `OrOptions`.
- `SelectPlayer`: pass `disabled: [{player, reason}]` in the 4th-arg options.
- `SelectCard` (card targets): pass `disabled: [{card, reason}]` in the config. Keep `cards` the SELECTABLE set — disabled cards ride the separate `disabledCards` model channel (server rejects them for free, the picker shows them behind the Available/All/Unavailable filter). Scope is **case-specific** (what's semantically relevant to THIS choice), not a count heuristic; only include cards the player can see (own/played/public — never hidden hands, deck, or other private zones).
- `SelectColony` (colony targets): set the input's `disabledColonies = [{colony, reason}]` (and `purpose`). Use `purpose: 'selectExistingColony'` (default) to make the client show ALL in-play colonies disabling the unpickable ones; `purpose: 'addNewColonyToGame'` for add-a-tile effects (only the offered not-in-play tiles show — never the existing colonies). Only send the reasons the client can't derive (TR affordability); the client already derives full / already-owned / visitor / no-fleet.
- The `reason` is an **English i18n key** describing the rule failure (the client can't always know the rule) — e.g. `'No plants to remove'`, `'Production already at minimum'`, `'Resources are protected'`, `'No Venus tag'`, `'No resources on this card'`, `'Cannot afford the TR increase to build here'`. Add the ru value to `ru/ui.json` — and reuse the project's CANONICAL term for any game concept (TR → **РТ**, VP → **ПО**, etc.; grep `ru/*.json` first, never coin a new abbreviation like "ПТ").

**Icon keys** are resolved in one place — `src/client/components/modalInputs/optionIcons.ts` `iconClassFor`. Three families, pick the right one for `icon`/metadata:
- standard resources: `megacredits` / `steel` / `titanium` / `plants` / `energy` / `heat` (the `Resource` enum values double as these keys).
- global parameters: `temperature` / `venus` / `oxygen` / `ocean`.
- card resources: the `card-resource-<key>` suffix (lowercase, spaces→hyphens — `addResourceToCard`/`removeResourceFromCard` derive it from the `CardResource` value).
- A genuinely new icon family → add it to the right set in `optionIcons.ts`; never hand-build the class string at the call site.

**Do NOT put in metadata what the client already derives** (avoid duplication / drift): per-player `current → resulting` values (computed from the public player models), corporation name, and the self-target flag are all client-side. The server only sends the *hint* (`icon` / `amount` / `scope`) and the *rule-based reason*.

**Localization rule (build crashes on dupes):** every new English key (verb buttonLabels, reasons, labels) goes in `ru/ui.json` (or `ru/play_prompts.json` for card-prompt titles) — but `grep` the EXACT key across ALL `src/locales/<lang>/*.json` first; `make_static_json` throws on a duplicate key in two files. If the existing translation of a shared key doesn't fit your context, introduce a NEW more-specific English key instead of reusing it.

**Quality gate:** open `?modalPlayground` (`ModalInputPlayground.vue`) and add/check a scenario for the new pattern. The per-scenario tags (`metadata: complete` / `rich target (production|stock)` / `disabled: N` / `fallback (text only)`) make a still-on-fallback prompt obvious. Dev tags live ONLY in the playground — never let `metadata complete`, `plain target`, English titles, or other technical strings leak into the real game UI.

**Current coverage / next frontier:** metadata + disabled candidates are done for player-target prompts (`SelectPlayer` + the OrOptions attack/steal deferreds), global-parameter, `SelectAmount`, card-resource options, **`SelectCard` (`RemoveResourcesFromCard`) with the Available/All/Unavailable filter**, and **`SelectColony` (`BuildColony` build-existing + Aridor add-tile + the trade flow)**. Not yet enriched (same model shape applies when you do): maxed global parameters, and the deliberately-skipped blue-card `action()` / production-phase prompts.

## Premium tooltips — NEVER the native `title` attribute

**The native browser `title=""` tooltip is BANNED in new UI.** It can't be styled to match the sci-fi chrome, it renders with an OS-default delay/look, and — critically — it's unreliable on `:disabled` controls (the very place we most need a "why is this disabled" hint). Use the shared **`.premium-tooltip(@max-width)`** LESS mixin (defined in `src/styles/card_action_buttons.less`, imported before every consumer) instead:

```less
.my-host { .premium-tooltip(220px); }
```
```html
<span class="my-host" :data-hint="reason">…</span>   <!-- reason '' → no tooltip -->
```

The mixin renders the reason from the host's `data-hint` attribute via a styled `::after` (dark glass + cyan rim), gated on `:hover` and `:not([data-hint=""])`. **Put `data-hint` on a NON-disabled host** — a disabled `<button>` never fires `:hover`, so wrap it (e.g. `.card-selection__action-wrap`, `.card-zoom-actions__tip`) and host the hint on the wrapper; the wrapper gets `:hover` from the pointer position even when the control inside is disabled. The mixin forces `position: relative`, so don't apply it to an element that must stay `position: absolute` (host the hint elsewhere or inline just the `::after`). Adopters so far: card-selection (per-card action button, confirm footer, fullscreen viewer buttons), colony tiles (select button, fleet badge), colony detail (status pills), colony trade-payment (disabled option cards). Migrate any remaining native `title` you touch.

**Known remaining exception:** the placement-lock tooltip in `PlayerHome.installPlacementGuards()` still sets a native `title` via JS on turn-ending buttons that mount mid-placement (a capture-phase `mouseover` attaches it to late-mounted buttons). That's a dynamic JS-driven case, not a static template binding; migrate it to `data-hint` + `.premium-tooltip()` when that subsystem is next touched.

**Disabled-button visuals:** never dim a button with whole-element `opacity` (it goes see-through over the board). The shared `.cab-base()` `:disabled` keeps the button OPAQUE (dim via `filter` + muted text); the CTA roles additionally override their bright plate to a solid dark "off" plate (`.cab-palette-cta()` `:disabled`). Follow the same "opaque, not transparent" rule for any new disabled control.

## Start-of-game flow — unified premium orchestration modal (`StartGameFlowOverlay`)

The whole start of generation 1 (play your 2 preludes one-by-one → apply the corporation's mandatory first action → confirm "begin the game") is driven by ONE App-level premium modal: `src/client/components/startGameFlow/StartGameFlowOverlay.vue`. It REPLACES, for the start sequence, both the bare prelude card-selection (`'Select prelude card to play'`, formerly `DraftFlowOverlay → CardSelectionContent`) and the corp first-action `OrOptions` (`'Take first action of X corporation'`, formerly `MandatoryInputModal → ModernOptionPicker`). Ordinary action modals stay focused — there is NO source-card / hero context injected into them (an earlier `StartEffectHeader` / `source`-threading experiment was fully rolled back, client AND server).

**Why App-level + module-state** (same rationale as `DraftFlowOverlay` / journal): `App.vue` mounts `<player-home :key="playerkey">` and bumps `playerkey` on every server response, remounting that subtree. The overlay is mounted as a SIBLING of `<player-home>` (in `App.vue`, next to `DraftFlowOverlay`, `:player-view` + `:waiting-on-players`), so it survives the remount and keeps the flow continuous across prelude → sub-action → waiting-for-others → corp → first turn. Its visibility latches sticky (see `startGameFlowActive`).

**Single source of truth = `src/client/components/startGameFlow/startGameFlowState.ts`** — owns the reactive `startGameFlowState` AND every predicate, so the overlay and the two legacy-suppression sites import ONE implementation. **State is keyed PER-PLAYER, not global:** `activatedPlayers` / `completedPlayers` are arrays of `view.id`, NOT single booleans. The module is shared across every player viewed in the SAME client (hot-seat / player switch), so a global `completed=true` (player 1 pressing "begin the game") would unmount/disable player 2's still-pending flow AND let the legacy corp `OrOptions` modal leak back in. `startGameFlowActive(view)` therefore checks `completedPlayers.includes(view.id)` / `activatedPlayers.includes(view.id)` — derived in the predicate (no watcher-reset race). `markStartFlowActivated(id)` / `markStartFlowCompleted(id)` mutate the arrays; `minimized` stays a single transient value (always reflects the currently-viewed player, re-derived by the overlay watcher each view change). The predicates:
**Detection is via an EXPLICIT server marker, never the (translatable) title** — see the i18n gotcha below for why. `BaseInputModel.startGamePrompt?: {kind: 'corporationInitialAction' | 'corporationSelection' | 'preludeSelection', preludeMode?: 'hand' | 'draw' | 'copy'}` (server: `BasePlayerInput.markStartGamePrompt(meta)`, set on the corp OrOptions in `Player.ts`, the SelectCard in `PreludesExpansion.selectPreludeToPlay`, and Merger's corp-select SelectCard in `Merger.bespokePlay`; serialized centrally in `ServerModel.getWaitingFor` next to `warning` / `pendingInitialActions`). The predicates:
- `startFlowPreludePrompt` / `startFlowPreludeDrawPrompt` / `startFlowPreludeCopyPrompt` — a `preludeSelection` SelectCard of mode `'hand'` (the player's own starting preludes → the grid) / `'draw'` (drew N, choose ONE, discard the rest — New Partner / Valley Trust → the draw-choice block) / `'copy'` (pick one ALREADY-PLAYED prelude to copy — Double Down → the copy block). `preludeMode` is computed authoritatively server-side: `'copy'` iff `cardAction === 'double-down'` (the ONLY reliable signal — Double Down's candidates are played preludes but can include New Partner from the hand, so a cards-vs-hand check misclassifies it), else `'hand'` iff `cards.every(c => preludeCardsInHand.includes(c))`, else `'draw'`. `startFlowAnyPreludePrompt` = any of the three.
- `startFlowCorpPrompt(view)` — an `OrOptions` whose `startGamePrompt.kind === 'corporationInitialAction'`. `corpActionOptionIndex(prompt)` returns the option without the `'pass'` warning (single-corp). For the MULTI-corp case (Merger gives a 2nd corp), `corpActionOptionIndexFor(prompt, corpName)` matches the option by its CARD token (the corp name, translation-proof) so each corp gets its own ПРИМЕНИТЬ ЭФФЕКТ. `corporationCardNames(view)` = all `CardType.CORPORATION` tableau cards (base + merged) in order; `corpStatusFor(view, name)` = `ready` (live OrOptions has an option for it) / `pending` / `done`. The corp area renders one COLUMN per corp with the apply button UNDER the card (so a second corp has room to its right).
- `startFlowCorpSelectPrompt(view)` — Merger's `corporationSelection` SelectCard (choose ONE of the dealt corps; unaffordable ones arrive with `CardModel.isDisabled` and render as a disabled "Недостаточно M€" hint). Picking submits `{type:'card', cards:[name]}`; the chosen corp joins the tableau (`playCorporationCard` → it shows in the corp area, and if it owes an initial action that flows through the corp OrOptions); the 42 M€ payment is a normal `SelectPaymentDeferred` sub-action. Suppressed in `DraftFlowOverlay` alongside the prelude prompts, and folded into `eligible` / `allDone` / `hasFocusedSubAction`.
- `preludeEntries(view)` — the player's STARTING prelude set for the grid, in a **STABLE order** (`preludeOrderStore`, a per-player append-only Map): a prelude's slot is fixed the first time it appears and NEVER reorders, so playing it only flips its status in place — it does NOT jump to the front (the grid `:key="entry.name"` keeps the element identity). played = `thisPlayer.tableau` filtered by `getCard(name)?.type === CardType.PRELUDE`; awaiting = `preludeCardsInHand`. ONLY drew-N-choose-ONE ('draw') candidates are EXCLUDED (`drawChoiceCandidateNames`, recorded at pick time) — they live in their own block. Double Down ('copy') candidates are NOT recorded and NOT excluded: they're already-played preludes that must STAY in the grid (the bug fixed here was treating a copy-source pick as a 'draw' choice, which deleted the source from the grid and broke the prelude counter). `playByName` dispatches a pick: draw → `playDrawPrelude` (records), copy → `playCopyPrelude` (submits, no record), else `playPrelude`. The order store is NON-reactive (it's written inside the `preludeEntries` computed; a reactive write would re-trigger it) and cleared on reset / completion. **Fizzle-block:** a playable prelude the server flagged `preludeFizzle` (it would do nothing now — e.g. Double Down before any other prelude is played, nothing to copy) gets `blocked: true` and shows a "Сначала разыграйте другой пролог" hint instead of РАЗЫГРАТЬ — BUT only while a non-fizzling playable prelude still remains (so the player is never trapped: when EVERY remaining prelude would fizzle, none are blocked and they can play one for the 15 M€). `blocked` names are also excluded from `playableZoomNames` (no fullscreen РАЗЫГРАТЬ).
- `startGameFlowEligible/Active/AllDone/HasFocusedSubAction` — gen-1 gating, the sticky-activated mount predicate (false during `isInitialDraftAwaiting` and after `completed`), the "begin the game" gate, and the "a focused sub-action is the prompt → collapse to the pill" predicate (which DOES still match the action-menu by title — that OrOptions TITLE, unlike the corp OPTION titles, is not mutated in place).

**Drew-N-choose-ONE block (New Partner / Valley Trust).** When `startFlowPreludeDrawPrompt` is live the overlay renders the drawn candidates in a distinct "ВЫБЕРИТЕ ОДИН ПРОЛОГ ЧТОБЫ РАЗЫГРАТЬ" block, each with РАЗЫГРАТЬ. On pick, `recordDrawChoice(view.id, candidates, chosen)` saves the decision into `startGameFlowState.drawChoices` BEFORE submitting (the server discards the non-chosen immediately, so they vanish from `playerView` — this is the only chance to capture them), then the resolved block shows the chosen РАЗЫГРАНА (green ✓) and the rest СБРОШЕНА (red ✕). Per-player; cleared on `markStartFlowCompleted` / reset.

**Two minimal server additions (the only server changes):** (1) `PlayerViewModel.pendingInitialActions: ReadonlyArray<CardName>` (self-model only, from `player.pendingInitialActions.map(c => c.name)`) — signals WHEN the corp still owes its first action (drives the corp-area "awaiting" status + the completion gate). (2) the `startGamePrompt` marker above. Both serialized in `ServerModel.getWaitingFor`/`getPlayerModel`; corp identity is already public so nothing leaks. Covered by `tests/models/ServerModel.spec.ts` + `tests/preludes/StartGamePromptMarker.spec.ts`.

**Routing suppression (so the legacy surfaces don't double-render):**
- `DraftFlowOverlay.cardInput` returns `undefined` when `startFlowAnyPreludePrompt(view)` matches (both hand + draw modes).
- `WaitingFor.useModalForCurrentInput` returns `false` when `startFlowCorpPrompt(view)` matches AND `startGameFlowActive(view)` (mirrors the existing `handPlayPrompt` suppression). The corp option's DEFERRED sub-actions (Tharsis city `SelectSpace`, Poseidon/Aridor `SelectColony`, payments) arrive as NEW top-level prompts AFTER the OrOptions is answered — they are NOT suppressed and render on their normal dedicated surfaces (`PlacementBanner` / `ColoniesOverlay` / `MandatoryInputModal`); the overlay just collapses to the pill via `startFlowHasFocusedSubAction`.

**⚠️ i18n MUTATES `Message.message` IN PLACE on render — never detect a prompt by its (translatable) title TEXT.** The shared `Message` object that backs an OPTION title has its `.message` field rewritten from the English template to the active locale the first time it's rendered (e.g. `'Take first action of ${0} corporation'` → `'Выполнить начальное действие корпорации «${0}»'`). Because `playerView.waitingFor` holds that SAME object, a predicate matching on the English text silently STOPS matching after the first render — which once leaked the legacy corp `OrOptions` modal back in (suppressed on the first computed pass, un-suppressed once the title was translated). **The fix is the explicit `startGamePrompt` marker** — a structural field i18n never touches. (An EARLIER fix detected via the title's CARD data token value; the marker superseded it as cleaner/production-grade.) When you add any new prompt detection, use a server marker or another structural signal, NEVER title text. The one surviving title check — the per-turn action menu in `startFlowHasFocusedSubAction` — is safe only because that OrOptions TITLE isn't mutated (the fork's inline action UI relies on the same `'Take your first/next action'` match and works). Covered by `tests/client/components/startGameFlow/WaitingForCorpSuppression.spec.ts` (mounts the real `WaitingFor` and asserts `useModalForCurrentInput === false` regardless of the title text).

**These predicates also run inside a computed, so they MUST NOT throw.** Every array access is guarded (`(view.pendingInitialActions ?? []).length`, same for `preludeCardsInHand`) so an older server that omits `pendingInitialActions` can't `TypeError` the computed and break the render.

**Submit + lifecycle:** the overlay copies `DraftFlowOverlay`'s App-level `onsave`/`applyPlayerViewUpdate` (POST `paths.PLAYER_INPUT`, shares `root.isServerSideRequestInProgress`, always does a full `playerkey++` remount — the overlay survives it). The backdrop is deliberately LIGHT and NOT blurred (`start_game_flow.less`) — a soft vignette, not a dark cover — so the player can read their panels and SEE the delta-chips that animate when a prelude / corp effect resolves.

**Fullscreen prelude play.** A single click on any prelude card opens it in the shared `CardZoomModal` (the card's built-in zoom is suppressed by `@click.capture.stop="openZoom(name)"` on the thumb). The `#actions` slot shows a РАЗЫГРАТЬ for PLAYABLE cards only (`playableZoomNames` = playable grid entries + live draw candidates); `playZoom` → `playByName(name)` dispatches to the draw recorder or the normal play and closes the viewer. `:cards="zoomNavCards"` (every shown prelude) gives ← / → navigation. (`playByName` is the extracted, test-friendly dispatch — the modal render is awkward under JSDOM teleport, so the spec exercises `playByName` + `playableZoomNames` directly.)

**Visual language (premium polish pass) — ONE unified status system across the whole modal**, all in `start_game_flow.less`; touch it, not per-state ad-hoc colours. The four card states read identically on preludes AND the corp: **AWAITING** (steel-blue, muted card), **ACT NOW** (cyan `0 0 0 1px`+glow frame with a slow `start-game-flow-card-glow` pulse — playable prelude / corp `--ready`), **DONE** (a small bright-green corner ✓ badge `.start-game-flow__played-check` [`--corp` = larger] + green frame — a played prelude OR a corp whose effect was applied, gated by `corpHadAction(name)` so a base corp with no start action stays badge-less), **DISCARDED** (grey card + red ✕ badge `.start-game-flow__discard-mark`). DONE/DISCARDED are CORNER badges, never centre discs — never cover the card art (readability rule). The corp area is the **hero**: `.start-game-flow__corp-item .start-game-flow__card-thumb` gets a focus frame + outer glow and `zoom: 0.66` (vs `0.6` for preludes). Status captions are pill **badges** with a leading dot (`.start-game-flow__corp-status--ready/pending/done`, `.start-game-flow__prelude-status--*`). Header is a row: hero gradient-clip title + subtitle (left) · progress **chips** (right) with a state dot (`--neutral/complete/ready/pending/done`). Tests: `startGameFlowState.spec.ts` (predicates + draw-choice) + `StartGameFlowOverlay.spec.ts` (render/submit/pill/begin/no-Pass/draw) + `WaitingForCorpSuppression.spec.ts` + `StartGamePromptMarker.spec.ts`.

**To extend the start flow** (a new corp/prelude needing input): if it's a corp first action or a prelude selection, it ALREADY carries the marker (set generically in `Player.ts` / `selectPreludeToPlay`) — nothing extra. ## Rematch flow (end-of-game — `RematchManager` + `RematchLayer`)

The premium end-of-game results overlay's old "New game" link is replaced by a **REMATCH** flow: a player offers a rematch, the others accept (or one declines), and once everyone has accepted a NEW game is auto-created with the SAME settings — then every player sees "Join rematch" (a private link to their seat in the new game). Built as a SELF-CONTAINED subsystem (a dedicated bounded route + an in-memory server singleton + an App-level client layer), exactly like the journal / notifications / endgame experience — it touches NEITHER `Game`/`SerializedGame` NOR `GameModel`/`PlayerViewModel`.

**Server (`src/server/rematch/`, `src/server/routes/ApiGameRematch.ts`):**
- `RematchManager` (singleton, `Map<GameId, RematchEntry>`, in-memory only — a rematch is decided while the finished game is still cached, so it is deliberately NOT serialized; a restart just drops the offer). Methods `offer` / `accept` / `decline` / `cancel` + `getModel(game, viewerId)`. `offer` seeds the offerer `accepted` and the rest `pending`; when ALL votes are `accepted` it calls `createRematchGame` and flips to `created`. A `creating` guard prevents a double-create when final accepts race. **Solo (1 player) creates immediately.**
- `createRematchGame` mirrors the relevant part of `ApiCreateGame` but sources everything from the finished game: same `gameOptions` (with `clonedGamedId` CLEARED — `Game.newInstance` THROWS on a set clonedGamedId), same players (name / colour / beginner / handicap) with fresh ids, same first player by colour, a NEW random seed (fresh board & decks), then `gameLoader.add(newGame)` (cache-add is enough to make it joinable — identical to the create-game route; it persists to the DB on first action).
- Route `GET/POST /api/game/rematch` (`paths.API_GAME_REMATCH`, registered in `requestProcessor.ts`). `GET ?id=<participantId>` returns the per-viewer `RematchModel` (works for players AND spectators so everyone can poll). `POST ?id=<playerId>&action=offer|accept|decline|cancel` is players-only (the action rides a query param, like `AcknowledgeDraw` — no body parsing).
- **Privacy:** the vote tally is public, but each player is told ONLY their OWN new-game participant id (`joinKind`/`joinId`, matched by colour) so nobody can open another player's perspective in the LIVE rematch (the finished-game ids are already public at END, but the new game is live). Spectators get the new spectator id.

**Client (`src/client/components/rematch/`):**
- `rematchState.ts` — module-level reactive (`model`, `submitting`) + `fetchRematch` / `submitRematch` / `rematchJoinHref` (builds `player?id=…` / `spectator?id=…`). Survives the `playerkey` remount like the other module states.
- `RematchLayer.vue` — App-level (mounted in `App.vue` next to `EndgameExperience`, gated by its OWN `v-if="endgameView !== undefined"` — placed BEFORE the endgame `v-if`/`v-else-if` screen chain so it doesn't break it). POLLS `/api/game/rematch` every 2 s while the game is over. Hosts: the **"Accept rematch?" prompt modal** (`RematchPromptModal.vue`, shown to every player who still owes a vote — the offerer never sees it; minimizable to a pill so the player can inspect results), a **"rematch ready" celebration** (Join / Watch link), and a **"X declined" toast** (auto-dismiss). This is the cross-player NOTIFICATION — independent of whether the results overlay is open.
- `EndgameResultsOverlay.vue` — the header control (where "New game" was) is now state-driven: **Offer rematch** → **Waiting N/M** (+ **Cancel** for the offerer) / **Accept** + **Decline** (for a voter, in addition to the global prompt) → **Join rematch** (player) / **Watch rematch** (spectator). A SPECTATOR with no active rematch keeps the original **New game** link.
- Styles `src/styles/rematch.less` (`@import` in `common.less` AFTER `endgame.less`; z 3300 modal / 3260 pill, above the results overlay's 3000 / reveal's 3200). i18n in `ru/ui.json` (do NOT re-add `New game` / `Minimize` / `Close` / `Cancel` — they already exist, `make:json` throws on dupes).

**Edge cases handled:** a DECLINE kills the offer (records `declinedBy`; anyone may offer again); only the offerer may `cancel`; a redundant `accept` / a fresh `offer` after `created` is a no-op; solo creates immediately; spectators can't POST. **The LEGACY `GameEnd.vue`** (reached only via a direct `/the-end?id=` link) intentionally keeps its "Create New Game" link — the premium results overlay is the canonical end-of-game surface and the only one with the rematch flow.

Tests: `tests/rematch/RematchManager.spec.ts` (the full state machine — offer/accept/decline/cancel, per-viewer join-id privacy, solo, double-create guard) + `tests/client/components/rematch/RematchControl.spec.ts` (the overlay control states). **To add a new participant action** extend the `RematchAction` union + the manager switch; the route + client submit are generic over the action string.

## Architecture

### Three-Layer Structure

- **`src/server/`** - Game engine, card logic, routes, database. Runs on Node.js.
- **`src/client/`** - Vue 3 frontend (Options API, `defineComponent`). Bundled with Webpack.
- **`src/common/`** - Shared types, enums, and models used by both client and server. No runtime logic that depends on either side.

The `@/` import alias maps to `./src/` (configured in tsconfig paths and webpack).

### Card System

Cards are the core domain object (~1000 cards across 15 modules). Each card involves:

1. **Card class** (`src/server/cards/<module>/CardName.ts`) - Extends `Card` (or `ActionCard` for cards with repeatable actions). Defines cost, tags, requirements, behavior, and metadata. Simple cards are purely declarative via the `behavior` property. Complex cards override `bespokePlay()`, `bespokeCanPlay()`, `bespokeCanAct()`, `bespokeAction()`.
2. **CardName enum entry** (`src/common/cards/CardName.ts`) - Every card needs an enum value here.
3. **Module manifest** (`src/server/cards/<module>/<Module>CardManifest.ts`) - Registers the card's factory in a `ModuleManifest`. Base cards use `StandardCardManifests.ts`. All manifests aggregate in `AllManifests.ts`.
4. **Card renderer** - Defined inline in the card's `metadata.renderData` using the `CardRenderer.builder()` DSL.
5. **Test** (`tests/cards/<module>/CardName.spec.ts`) - Uses `testGame()` and `TestPlayer` helpers.

Card types: `EVENT`, `ACTIVE` (has action), `AUTOMATED`, `PRELUDE`, `CORPORATION`, `CEO`, `STANDARD_PROJECT`, `STANDARD_ACTION`.

**Card vs ActionCard**: Extend `Card` for automated/event cards. Extend `ActionCard` for ACTIVE cards with a repeatable action — `ActionCard` enforces that an `action` behavior is defined and provides `canAct()`/`action()` wiring automatically.

### Behavior System

The `Behavior` type (`src/server/behavior/Behavior.ts`) is a declarative DSL for card effects: production changes, resource gains/losses, tile placement, TR changes, global parameter increases, drawing cards, etc. Cards set `behavior` (on play) and/or `action` (repeatable) properties. The `BehaviorExecutor` (`src/server/behavior/Executor.ts`) interprets these at runtime. Prefer declarative `behavior` over imperative `bespokePlay()` overrides when possible.

### Deferred Actions

Player choices and multi-step effects use `DeferredAction` (`src/server/deferredActions/`). Actions are queued via `game.defer(action)` with a `Priority` and resolved in order. The `.andThen()` callback chains follow-up logic after a deferred action resolves.

### Player Inputs

When a player needs to make a choice, the server returns a `PlayerInput` (e.g., `SelectSpace`, `SelectCard`, `OrOptions`). These live in `src/server/inputs/`. The client renders the appropriate UI based on the input type.

### Game Modules (Expansions)

Each expansion has its own directory under `src/server/cards/` and a manifest. Source directories: `base`, `corporation` (Corporate Era), `promo`, `venusNext`, `colonies`, `prelude`, `prelude2`, `turmoil`, `community`, `ares`, `moon`, `pathfinders`, `ceos`, `starwars`, `underworld`, `delta`. Test directories under `tests/cards/` mostly mirror these (note `tests/cards/ceo` vs `src/server/cards/ceos`). Cross-expansion card compatibility is declared via `compatibility` in `CardFactorySpec`.

### Client Components

Vue 3 with Options API. Components are in `src/client/components/`. The root `App.ts` routes between screens. `PlayerHome.vue` is the main game view. Card rendering components are in `src/client/components/card/`. Styles use Less (`src/styles/`).

### Database

Pluggable backends in `src/server/database/`: `SQLite`, `PostgreSQL`, `LocalFilesystem`. Games are serialized/deserialized through `SerializedGame`/`SerializedPlayer` types. `GameLoader` handles caching and retrieval.

### Testing Patterns

- **`testGame(n, options?)`** (`tests/TestGame.ts`) - Creates a game with n players, returns `[game, ...players]` as a tuple. Skips initial card selection and disables Ares hazards by default.
- **`TestPlayer`** (`tests/TestPlayer.ts`) - Extends `Player` with test utilities like `popWaitingFor()`.
- **Test utilities** (`tests/TestingUtils.ts`) - Key helpers: `runAllActions(game)` (process deferred action queue), `cast(value, class)` (type-safe cast for PlayerInputs), `setOxygenLevel()`, `setTemperature()`, `addOcean()`, `addGreenery()`, `addCity()`, `fakeCard()`.
- Server card tests: instantiate the card, call `canPlay()`/`play()`/`action()`, assert state changes. Call `runAllActions(game)` after actions that queue deferred actions.
- Client tests: use `@vue/test-utils` mount/shallowMount with JSDOM setup from `tests/client/components/setup.ts`.
- Test framework: Mocha + Chai (expect style). Client tests use mochapack.

### Internationalization

Custom i18n via `src/client/directives/i18n.ts` with `v-i18n` directive. Translation files in `src/locales/`. Strings are matched by exact text content.

**Russian (ru) translations — terminology consistency.** Before inventing a new Russian translation for a term, check two sources, in order:

1. **The project itself.** Grep `src/locales/ru/` for the term (or close variants — singular/plural, case-different) and reuse the existing wording. Examples already established: `Building → Здание`, `Space → Космос`, `Science → Наука`, `Power → Энергия`, `Earth → Земля`, `Venus → Венера`, `Plant → Растение`, `Microbe → Бактерия`, `Animal → Животное`, `Event → Событие`, `Wild → Любая`, `VP → ПО`, `researching → исследование`, `passed → спасовал`. Iconography terms live in `help_iconography.json`; UI labels in `ui.json`; log templates in `log_messages.json`; game-end strings in `game_end.json`.
2. **The official Russian edition of the *Terraforming Mars* board game** (Crowd Games / «Покорение Марса»). Game-specific terms — tag names, milestone/award names, standard project labels, resource names — should match the printed Russian cards. If a project translation contradicts the board-game canon, surface the discrepancy to the user rather than silently picking one.

Never translate proper nouns that look like player names or English card names already in the player log unless they appear in a translation file.

**NEVER modify the Russian translation of an English key you didn't introduce yourself.** The same English string can appear in many places — log messages, card descriptions, tooltips, UI labels — each with its own context. Changing `"Convert" → "Превратить"` to make your new button read nicely will silently rewrite "Convert" everywhere else in the game (action logs, card flavor text, etc.) and break the meaning. Instead, **introduce a new English key** for your UI element (e.g. `"Spend"` or `"Convert plants action button"`) and add its translation. If you really must reuse an existing key, first `grep` every usage of that English string in `src/client/`, `src/server/` and the other `src/locales/<lang>/` files, confirm the new wording fits ALL of them, and call it out in your summary.

**Before adding a translation key, `grep` for the EXACT key across ALL files in `src/locales/<lang>/` to make sure it doesn't already exist.** The static-json build tool (`src/tools/make_static_json.ts:85`) aggregates every JSON file under a language directory into a single dictionary and **throws on duplicate keys** with `ru: Repeated translation for [...]`. The duplicate can be in `ui.json`, `UI_cards.json`, `game_end.json`, `log_messages.json`, `help_iconography.json`, etc. — same key in two files crashes the build. Run e.g. `grep -nE '^\s*"card":' src/locales/ru/*.json` for every key you intend to add. If a match exists and its translation already fits your use, reuse it (don't add a duplicate). If the existing translation doesn't fit, pick a different more-specific English key for your context.

### Centering UI under a `<Card>`

`.card-container` carries a legacy **asymmetric** margin from `src/styles/cards.less:90`: `margin: 15px 30px 10px 0px`. This was originally for inter-card spacing in grid layouts (OtherPlayer played-cards row, SortableCards hand, etc.).

The 30-px right margin makes a wrapping element's bounding box **30 px wider than the visible card**. The card silhouette sits flush-left within that box; the right 30 px is empty space. **Any centered UI placed under the card (in a flex column / grid column whose width is driven by the card)** will land ~15 px right of the card's visual centre because the parent centers the wider bounding box, not the visible card.

**Rule:** when you mount custom UI (buttons, badges, status chips) below a `<Card>` in a flex column / grid column and expect it to read as centered on the card silhouette, **zero out `.card-container`'s margin in your wrapper's scope**:

```less
.your-slot-wrapper > .card-container {
    margin: 0;
}
```

Don't try to compensate by shifting the UI manually (e.g. `margin-left: -15px`) — `.card-container`'s margin changes with zoom (the media-query ladder in `card_selection.less` scales `.card-container` between 1.0 and 0.68) and the offset would drift across breakpoints.

The reference implementation is `.card-selection__card-clickable > .card-container { margin: 0; }` in `src/styles/card_selection.less`. Mirror that pattern for any new flow that stacks UI under a Card (e.g. future hand-card "РАЗЫГРАТЬ" button, opponent-tableau action buttons).

Inter-card horizontal spacing in your grid should be owned by the parent container (`gap`, `grid-column-gap`, etc.) — independent of `.card-container`'s own margin, which only matters for legacy non-flex layouts.

### Logging

Game actions are logged via `game.log()` and `player.log()` which accept template strings with `${player}`, `${card}`, `${amount}` style placeholders and corresponding `LogMessageData` entries. Log statements appear in the game's action log visible to players.
